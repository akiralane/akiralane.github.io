The Parser

We start by defining what a parser is:

newtype Parser a = P (String -> [(a, String)])

That is, a parser is a function which takes a string as its argument and returns a list of results.

Each result is a pair:

The first component is a value of type a produced by parsing a prefix of the string.

The second component is the unparsed suffix of the argument string.

Intuitively, the empty list denotes the failure of a parser (where the input string does not conform to the described grammar).

Returning a list of results allows us to build parsers for ambiguous grammars, since any one of the results is a valid interpretation.

Making Useful Functions

Using our newly defined concept of a parser, we can begin to build useful functions which do simple parsing.

item

We can start by simply extracting a single item from the input string:

item :: Parser Char
item = P (\inp -> case inp of
										[] -> []
                    (x:xs) -> [(x,xs)]
          )

This fails (returning an empty list) if the input string is empty and consumes the first character of the input string otherwise.

parse

parse simply takes a parser and a string and feeds the string as input to the parser to retrieve its results.

parse :: Parser a -> String -> [(a, String)]
parse (P parser) input = parser input

alternative (<|>)

This operator takes two parsers and tries parsing using the first. If that fails, it tries parsing using the second.

(<|>) :: Parser a -> Parser a -> Parser a
p <|> q = P (\input -> case parse p input of
													[] -> parse q input
													[(v,out)] -> [(v,out)]
						)

Sequencing Parsers

In the same way as IO actions, functions of type Parser are not easily composed...

However, also in the same way as IO actions, the do notation works fine for this!

Remember the firstAndThird example from a little while back?

firstAndThird :: Parser (Char, Char)
firstAndThird = do x <- item
									 item -- y
									 z <- item
									 return (x,z)

do for Parsers

The rules for the do notation apply just the same here.

Each action must be column-aligned, you don't have to bind every result, you can nest blocks, etc.

Remember that the result of the last action is the return value for the whole block.

Deriving Parsing Primitives

Using what we have, we can now build a library of parsing primitives by sequencing parsing actions, just as we built a library of IO actions.

A good starting point is the sat parser function, which takes a single character and parses or not based on whether the character read satisfies a given predicate.

sat :: (Char -> Bool) -> Parser Char
sat p = do x <- item
					 if p x then
							return x
					 else
							P (\input -> [])
digit :: Parser Char
digit = sat isDigit
char :: Char -> ParserChar
char x = sat (x ==)

Parsing Repeatedly (+, *)

Consider how we might parse a string of lowercase letters:

stringLower :: Parser String
stringLower = do x <- (sat isLower)
                 xs <- stringLower
                 return (x:xs)
							<|> return []

We effectively either parse zero characters or parse one and then recursively parse more.

We can generalise this repeated parsing to more than just Strings.

many :: Parser a -> Parser [a]
many p = many1 p <|> return []


many1 :: Parser a -> Parser [a]
many1 p = do v <- p
             vs <- many p
             return (v:vs)

many1 is typically called some.

Examples

Here are some examples of usage!

nat :: Parser Int
nat = do ds <- some digit
         return (read ds)


int :: Parser Int
int = do char '-'
				 n <- nat
	       return (-n)
      <|> nat

Here's one for parsing Haskell-style lists of digits:

p :: Parser String
p = do char '['
       d <- digit
       ds <- many (do char ','
                      digit)
       char ']'
       return (d:ds)
-- > parse p "[1,2,3,4]"
-- [("1234", "")]

-- > parse p "[1,2,3,4"
-- []

Example Grammar for Arithmetic Expressions

Definition

In order to implement the associativity and tighter binding of * we have to rewrite the concrete grammar of arithmetic expressions as follows:

expr ::= term '+' expr | term

term ::= factor '*' term | factor

factor ::= nat | '(' expr ')'

nat ::= digit | digit nat
digit ::= '0' | '1' | ... | '9'

Note how an expr is either a term '+' expr or a term. Since + has lower precedence than *, it should appear higher in the parse tree.

Notice the brackets in factor and how they contain an expr - this is effectively resetting the precedence by starting back at the top of the subtree.

Implementation

expr :: Parser Int
expr = do t <- term
          char '+'
          e <- expr
          return (t + e)
       <|> term
term :: Parser Int
term = do f <- factor
          char '*'
          t <- term
          return (f * t)
       <|> factor
factor :: Parser Int
factor = nat <|> do char '('
                    e <- expr
                    char ')'
                    return e
nat :: Parser Int
nat = do ds <- some digit
         return (read ds)

-- (defined earlier)

Finally, we define the entry point to the parser:

parseAndEval :: String -> Int
parseAndEval xs = fst (head (parse expr xs))

-- > parseAndEval "2*3+4"
-- 10

-- > parseAndEval "2*(3+4)"
-- 14
