Defining Classes

We've seen classes in Haskell and how they specify interfaces which must be implemented.

We have also seen that types can be instances of many classes - Show, Read, Eq, Ord, Num, etc.

We can define our own classes using the class keyword.

You need to give the class a name and list types of the functions required of instances.

You are allowed to put constraints on the class itself.

You can provide default implementations of functions as well.

Example

Here's how the Ord class is defined:

class Eq a => Ord a where
	(<),(<=),(>),(>=) :: a -> a -> Bool
	min, max :: a -> a -> Bool
	min x y | x <= y = x
				  | otherwise = y
	max x y | x <= y = y
          | otherwise = x

Eq a is a class constraint - members of Ord must first be members of Eq.

(<),(<=),(>),(>=) are undefined functions - it is up to the instance to decide how to implement them.

min, max are given default implementations. However, notice how these implementations depend on one of the as-yet undefined comparison operators. This is fine; when the instance provides an implementation of <= (which it has to), min and max will become well-defined.

Generally:

class CONSTRAINTS => NAME a where
	function :: type
	function :: type
	...
	function = default definition
	function = default definition

Declaring instances

To declare an instance, we use the instance keyword. When you do this, you must also provide implementations for all the undefined functions.

Here's Bool:

instance Ord Bool where
	False < True = True
	_     < _    = False
	b <= c = (b < c) || (b == c)
	b >  c = (c < b)
  b >= c = c <= b

Deriving instances

For new types which are built in a structurally simple way, Haskell can automatically derive implementations of the basic classes.

This, unsurprisingly, is done using the deriving keyword. It can save a lot of work when default implementations are fine.

data Day = Mon | Tue | Wed | Thu | Fri | Sat | Sun
	deriving (Eq, Ord, Show, Read)

The above tells Haskell to work out the implementations of the functions in all of those classes by simply looking at the structure of Day.

For example, show will simply print out the string name of any given value.

< from Ord is defined such that Mon < Tue < ... < Sun.

Deriving with parameters

For type declarations with parameters, if the type is declared to derive a class, then all of its parameters must derive that class as well.

-- fine because Float does all of these already
data Shape = Circle Float | Rect Float Float
	deriving (Eq, Ord, Show, Read)

-- !! NOT fine because (a -> a) is not an instance of Eq
-- hence, no default implementation of == exists
data DocumentedFunction a = Doc String (a -> a)
	deriving Eq
