Type synonyms

As you know, String is a synonym for [Char]. Then, what's the point of String?

It provides better documentation of the type of a function - it helps to indicate intended usage.

Haskell lets you build your own synonyms:

type Pos = (Int, Int)

-- this makes concepts relating to positions easier to understand
origin :: Pos
origin = (0,0)

moveLeft :: Pos -> Pos
moveLeft (x,y) = (x-1, y)

Parameterised type synonyms

Typically, we use type synonyms as a means of describing an assembly of types. For example, an association is a list of pairs of keys and values (like a Python dictionary, but a list).

To allow creating these, Haskell allows type synonyms to carry parameters:

type Association k v = [ (k,v) ]

find :: Eq k => k -> Association k v -> v
find k a = head [ v | (k', v) <- a, k==k']

These can't be partially applied; you can't create an IntAssoc type (from Int to a) by feeding an Int to Association.

Nested and recursive type synonyms

Nested type synonyms are allowed - that is, you can define type synonyms in terms of other synonyms.

Recursive type synonyms are not allowed.

Making new types

Note that you're not creating a new type when you use the type keyword. It really is just a synonym. This means that our moveLeft function from above will also accept any pair, even if it's not explicitly declared as a Pos.

What if I wanted to create a new type which behaved a lot like an Int, but I don't want functions on the type to be able to accept an Int instead?

For example, if I have introduced some sort of guard when creating this new type which guarantees something about its value - like a bounded int, or even numbers only.

Keyword data

The data keyword lets us do just this - we can define a new type which is not just a new name for an existing type. But first, let's talk about constructors.

Simple Constructors

A constructor is just a name tag which identified the values in a type.

These can be nullary, which means the name is all there is:

Bool contains the nullary constructors True and False. There's no data on top of those names.

Constructors may also take one or more arguments - this is how we build larger values in the type.

Using data

Here's what a unary constructor looks like:

data Foo = F Int

:type F 5 in GHCI will tell us that F 5 :: Foo.

We have the constructor F for the type Foo which can take an Int.

Of course, we can also use the keyword to create entirely new finite types where every value is listed explicitly - somewhat like an enum in other languages.

data Day = Mon | Tue | Wed | Thu | Fri | Sat | Sun

isWeekend :: Day -> Bool
isWeekend Sat = True
isWeekend Sun = True
isWeekend _   = False

isWeekday :: Day -> Bool
isWeekday = not . isWeekend

Constructors with multiple parameters

As you'd expect, constructors can take more than one parameter.

data Shape = Circle Float | Rect Float Float

-- | Create an equal-sided Rect from a float
square :: Float -> Shape
square n = Rect n n

-- | Find the area of a shape
area :: Shape -> Float
area (Circle r) = pi * (r^2)
area (Rect x y) = x * y

newtype

Declaring a brand new type for something like our Foo = F Int has a computational cost, because every function which uses it must pattern match F and extract the Int from it, then build it back up to get back to the Foo value.

What if we could be as efficient as just using an Int, but retain the type safety aspect? Here's a slightly obscure alternative: newtype.

It can only be used for types of a single unary constructor.

newtype Foo = F Int

Foo is now not interchangeable with Int, but also the compiler optimises away the pattern matching to improve efficiency!

This is a bit of an edge case, so don't worry too much about it.

Maybe

You know Rust's Option? It's in Haskell! (or rather Haskell's Maybe is in Rust...)

data Maybe a = Nothing | Just a

This lets us handle error cases more gracefully, as it does in Rust.

Recursive types

Recursive types are fine as long as you're using data.

data MyList a = Empty | Cons a (MyList a)

What quickly becomes clear when dealing with these is that pattern matching works on constructors - even user defined ones.

This makes for a powerful mechanism for defining new types and functions over them.

For example, it's pretty easy to build a datatype of Binary Trees.

data Tree a = Leaf a | Node (Tree a) a (Tree a)

The node contains a subtree to the left, the data it contains, and then a subtree to the right. Each subtree can actually be a Leaf.
