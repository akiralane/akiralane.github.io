We saw in previous topics that we could declare different data types for trees:

data LTree a = Leaf a | node (LTree a) (LTree a)
data MTree a = Node a [MTree a]

The trees that these produce have different shapes - binary trees, etc.

Algebraic data types are incredibly useful in this respect, in that we can specify the shapes of trees very carefully.

Red-black trees

We also see another benefit from Haskell's data type declarations that is perhaps less obvious.

Consider the well-know red-black tree data structure. Remember the constraints:

  • The root is black
  • All leaves are black
  • If a node is red, its children are black
  • Every path from any node to a leaf contains the same number of black nodes.

Red-black trees as a sucky datatype

type Colour = String
data RBTree a = Leaf a Colour | Node a Colour (RBTree a) (RBTree a)

Seems alright at first glance, but the colour being encoded as a String is obviously bad. The compiler can't detect typos or whatever in strings!

Also, it forces us to use guards in our functions on the tree, which are often prone to logical mistakes.

Red-black trees as a not-sucky datatype

data RBTree a = Leaf a
              | RedNode a   (RBTree a) (RBTree a)
              | BlackNode a (RBTree a) (RBTree a)

Constructors are much better. Ah, how we love to do pattern matching.

This concept is very useful when it comes to modelling more complex tree types.

For example, let's have a look at abstract syntax trees for programming languages.

Abstract Syntax Trees

ASTs are trees which represent the structure of a program expression and remove unnecessary syntactic information, like redundant brackets.

Different node types can represent different types of expression directly in the tree structure.

This makes them an excellent data structure for then defining functions of programs (such as an interpreter!) over.

AST for arithmetic expressions

Given that Haskell data type definitions look a lot like grammars, we can use these datatypes to represent the abstract syntax trees of a programming language.

Let's try this for a simple language of arithmetic expressions:

data Expr = Val Int | Add Expr Expr | Sub Expr Expr

Looks a lot like a tree and a grammar, doesn't it? Makes sense.

The values of this data type are ASTs for the language. We can then write the interpreter function to evaluate expressions very easily:

eval :: Expr -> Int
eval (Val n) = n
eval (Add e1 e2) = eval e1 + eval e2
eval (Sub e1 e2) = eval e1 - eval e2
