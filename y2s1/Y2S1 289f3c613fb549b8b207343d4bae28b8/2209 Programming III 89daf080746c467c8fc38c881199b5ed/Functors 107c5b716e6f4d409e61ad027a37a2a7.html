Higher-order functions on trees

As you'd expect, we can make use of higher-order functions over structured data.

The examples of higher-order functions we have seen already have mostly been concerned with list processing: map, filter, fold.

There's no reason we can't extend these ideas to other, more complex structures!

Tree maps

Suppose we want to take a tree of Strings and map the function upper across all the nodes in the tree. We could write a recursive function using pattern-matching to do this, but we could also recoginise that this function would fit a well-known template of recursion we know loosely as map.

We can do this fairly easily for the trees we've given as examples so far:

data LTree a = Leaf a | Node (LTree a) (LTree a)
lTMap :: (a->b) -> LTree a -> LTree b
lTMap g (Leaf x) = Leaf (g x)
lTMap g (Node l r) = Node (lTMap g l) (lTMap g r)

data ITree a = Leaf | Node a (ITree a) (ITree a)
iTMap :: (a->b) -> ITree a -> ITree b
iTMap g (Leaf) = Leaf
iTMap g (Node x l r) = Node (g x) (iTMap g l)(iTMap g r)

data MTree a = Node a [MTree a]
mTMap :: (a->b) -> MTree a -> MTree b
mTMap g (Node x ts) = Node (g x) (map (mTMap g) ts)

Introducing functors

I'm going to make the functor-function typo a lot, aren't I?

Notice that the type of the map function in each case above is pretty much the same, except for the name of the Tree type itself:

xyzMap :: (a->b) -> xyzTree a -> xyzTree

This suggests that we could formulate the map function in a class of mappable types. This would give us a single consistent name for the map function.

The above is known as a functor.

In essence, a functor is a class which represents the types for which it makes sense to have a map function. For example, lists, trees and similar structure can support a map function, where a function is applied to every point in the structure.

This works because fundamentally, a map is a structure-preserving operation.

The functor class

It's a simple class, with one function fmap to implement.

class Functor f where
	fmap :: (a->b) -> f a -> f b

Here are some instances:

instance Functor [] where
	fmap = map

instance Functor MTree where
	fmap = mTMap

-- oh!
instance Functor Maybe where
	fmap _ Nothing = Nothing
	fmap g (Just x) = Just (g x)

What's the point?

The advantage is that, for any type you know is an instance of Functor, you know the name of its map function. It should also have predictable behaviour (although there's no guarantee of this).

Further, it provides a useful abstraction to help with polymorphism:

altMap :: Functor t => (a -> b) -> (a -> b) -> t a -> t b

This lets me say that t is any type which can support a map.

Expected properties of fmap

It is expected (though not enforced) that the following equations should hold for any implementation of fmap.

-- functor law 1
fmap id = id

-- functor law 2
fmap (g.h) = fmap g . fmap h
