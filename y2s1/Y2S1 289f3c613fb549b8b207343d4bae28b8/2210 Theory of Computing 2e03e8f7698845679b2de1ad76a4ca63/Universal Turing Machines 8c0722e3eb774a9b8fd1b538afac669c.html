Universal Turing Machines

We just saw in the last topic that there are TMs which can simulate other TMs. What if we went further, and found a TM which could simulate any other TM...?

This isn't a theoretical exercise - we use UTMs every day. In terms of programming languages, it's an interpreter - a program which takes in a program and simulates the computation!

The idea is to find a suitable encoding of TMs to construct a TM UUU﻿ such that:

L(U)={M#x∣x∈L(M)}L(U)=\{M\#x|x\in L(M)\}L(U)={M#x∣x∈L(M)}﻿

The M#xM\# xM#x﻿ looks strange; here's what it means:

  • MMM﻿ is an encoding of the machine we want to simulate. We'll look at this in more depth later.
  • #\##﻿ is a separator symbol.
  • xxx﻿ is an encoding of the input string.

Our UTM works roughly as follows:

  1. Check whether MMM﻿ and xxx﻿ are valid encodings - reject if not.
  1. Simulate MMM﻿ for input xxx﻿.
  1. Accept or reject depending on whether MMM﻿ accepts or rejects.

There are a bunch of ways to encode Turing machines - the way that it's done is not really significant. Here's an example anyway, though, using the alphabet {0,1}\{0,1\}{0,1}﻿:

This uses 111﻿ as a separator between groups of 000﻿'s which actually describe the machine.

  • states {0,1,…,n−1}\{0,1,\dots,n-1\}{0,1,…,n−1}﻿ then a separator,
  • tape alphabet {0,1,…,m−1}\{0,1,\dots,m-1\}{0,1,…,m−1}﻿ of which the first kkk﻿ is the input alphabet, then a separator,
  • the start, accept and reject states s,t,rs,t,r s,t,r﻿ each followed by separators,
  • the blank symbol ⊔\sqcup⊔﻿ and endmarker ⊢\vdash⊢﻿ each followed by separators.

Constructing a universal Turing machine

Once we've found a suitable encoding, we can start constructing our machine

L(U)={M#x∣x∈L(M)}L(U)=\{M\#x|x\in L(M)\}L(U)={M#x∣x∈L(M)}﻿

So, how does it work?

The tape is paritioned into three tracks:

  1. TOP: description of MMM﻿
  1. MIDDLE: contents of MMM﻿'s tape
  1. BOTTOM: MMM﻿'s current state and position on its tape

For each step, the universal machine:

  1. looks at MMM﻿'s current state and head position from the bottom tape
  1. reads the corresponding tape contents at the correct position from the middle tape
  1. reads the relevant transition from the top tape
  1. simulates the transition, updating the tape, state and head.
  1. accepts if MMM﻿ accepts, rejects if MMM﻿ rejects.
  1. otherwise, continues.
