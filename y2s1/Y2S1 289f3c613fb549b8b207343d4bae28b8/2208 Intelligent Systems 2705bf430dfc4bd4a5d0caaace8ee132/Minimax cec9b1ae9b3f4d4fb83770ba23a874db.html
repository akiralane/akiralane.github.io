Summary

Search problems so far have concentrated on unopposed search problems (8-square, 8-queens etc.).

However, game playing is a traditional focus for classical AI. In this, we must take account of the opponent.

In a game tree search, we have the following, which is fairly self-evident:

Initial state is the initial state of the game.

Goal state is the set of winning positions.

Successor function generates the legal moves from this state.

Evaluation function assigns a score to each state.

Minimax search

Minimax is a view of search in adversarial spaces.

MAX is traditionally the player, and picks the moves whose heuristic value is the best (or maximum). MIN is the opponent, and picks moves which minimise MAX’s advantage.

Since we’re going through a tree, typically turn-by-turn, each layer of the tree is assigned to the players in an alternating manner.

We search to a given ply (depth-limited search), and evaluate heuristic for leaf nodes.

The internal nodes propagate the values of the heuristic back towards the root. This means that MAX nodes can take the maximum of their child values, while MIN nodes take the minimum of their child values.

Properties of minimax

Complete? Yes, if the tree is finite.

Optimal? Yes, against an optimal opponent.

Time complexity of O(bm)O(b^m)O(bm)﻿.

Space complexity of O(bm)O(bm)O(bm)﻿ (from depth-first).

Evaluation functions

Typically, the evaluation function is a linear function in which coefficients are used to weight game features. For most games, it’s unlikely to be a perfect, computable function.

Games with uncertainty - like dice games - add notions of expectation.

The horizon effect

We can’t exhaustively search most game trees, obviously.

Significant events might exist just beyond the part of the tree we’ve searched. What if our opponent is looking n+1n+1n+1﻿ moves ahead?

Since only a partial game tree has been analysed, an inevitable event can appear preventable to an algorithm.

How do we counteract this?

Quiescent search

A bad evaluation function can lead to wild swings in behaviour (look at simply evaluating chess on material balance). However, interesting things can also happen where these wild swings occur.

Quiescent search attempts to emulate the human intuition of looking deeper at interesting moves - we instruct our algorithm to search “volatile” positions than “quiet” ones to make sure there aren’t any hidden traps etc.

Singular extension

Explore more down one branch if one move is significantly better than the others, or in cases where there are fewer legal moves.

Example: The Game of Nim

In Nim, two players start with a single pile of seven matches. Each player takes it in turn to take a pile of matches and split it into two differently-sized piles of matches.

The last player who is able to make a move is the winner.

It’s a very small game, with few legal states and few legal moves from each state. It’s very possible to draw out a tree representing all states of the game, and find that the player who plays second has an unfair advantage...

Minimax for Nim

We use an evaluation function that reports:

+1 if MAX has a winning move

-1 if MIN has a winning move

Summary

Evaluation functions are core to adversarial games. They’re usually a weighted linear function of easily computed game attributes.

Minimax is the de-facto algorithm to solve them.

Ply depth is very important.

Minimax isn’t efficient!
