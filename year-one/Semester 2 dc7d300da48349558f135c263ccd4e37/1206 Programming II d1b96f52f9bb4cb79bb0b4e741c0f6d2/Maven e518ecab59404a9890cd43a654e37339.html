Maven

Maven is a package manager which handles all of the dependencies and horrible messy stuff for us (see notes on JAR files).

All of the config for Maven is kept in the pom.xml file.

The easiest way to start up a pom and a pre-set-up default configuration is by using Maven's quickstart archetype:

mvn archetype:generate -DarchetypeArtifactId=maven-archetype-quickstart -DarchetypeVersion=1.4

Now, we can tell Maven which dependencies we want. The XML to add to the pom can usually be found with a quick Google search. Example:

<dependencies>

	<dependency>
		<groupId>org.junit.jupiter</groupId>
		<artifactId>junit-jupiter-engine</artifactId>
		<version>5.7.0</version>
		<scope>test</scope>
	<dependency>

	<!-- etc... -->

</dependencies>

Now we'd like Maven to make the JAR for us.

Just like with the JAR file, we need to tell Maven where our main class is. Unlike with the JAR, it doesn't suck ass this time. We'll want the maven-jar-plugin.

IF IT'S NOT THERE ALREADY, PLEASE FOR THE LOVE OF GOD DON'T PUT IT IN THE PLUGINSMANAGEMENT SECTION. MAKE A NEW PLUGINS SECTION IF YOU HAVE TO.

<plugins>
  <plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-jar-plugin</artifactId>
    <configuration>
      <archive>
        <manifest>
          <addClasspath>true</addClasspath>
          <mainClass>point.me.toThe.App</mainClass>
        </manifest>
      </archive>
    </configuration>
  </plugin>
</plugins>

<plugins>
	<pluginManagement>
		<!-- PLEASE NOT HERE -->
	</pluginManagement>
</plugins>

If we want to be able to run our application with Maven (which is v. useful) we can use the maven-exec-plugin. Put this in the plugins section and configure it for our main class.

mvn clean removes any leftover from previous builds.

mvn install builds all of our java source.

mvn exec:java runs our program.

We can do all of these at once if we like: mvn clean install exec:java.

Distribute Option 1: UBER JAR!!!!!!!!!

Maven's output (your freshly baked jar etc.) goes into the target folder.

It's fine for running with mvn exec:java but if we try to run it straight with java, we run into the problem of it not finding our libraries again.

We can use maven-shade-plugin to build our uber JAR and shade it if that's what we want. 

<plugin>
	<artifactId>maven-shade-plugin</artifactId>
	<version>3.2.4</version>
	<executions>
		<execution>
		<goals>
			<goal>shade</goal>
		</goals>
		<configuration>
			<shadedArtifactAttached>true</shadedArtifactAttached>
			<!-- the above makes a separate shaded JAR instead of replacing the original -->
			<transformer implementation=
				"org.apache.maven.plugins.shade.resource.ManifestResourceTransformer">
				<mainClass>point.me.toYour.App<mainClass>
			</transformer>
		</configuration>
		</execution>
	</executions>
</plugin>

Distribute Option 2: [own the] lib[s] folder

This option is better if you want your libraries separate - for building multiple apps with the same libraries, or making a library yourself.

Use the maven-dependency-plugin, which will put all of our dependencies into a lib folder for us.

We then need to edit the maven-jar-plugin so that it writes the correct classpath to MANIFEST.MF.

Between the <manifest></manifest>tags, add <classpathPrefix>lib</classpathPrefex>.

mvn clean install will now give us a nice runnable jar, which points to a lib folder for its dependencies.

Tests

You can drop tests into src/test and run them with mvn test. These also get run every time you try to build and will stop a build on a failed test. This is pretty useful!

List of oopsies

  • In your pom, is every tag closed properly? You might have missed a /.
  • Make sure there's a newline at the end of Manifest.txt, although this shouldn't matter with Maven.
  • Make sure source code structure, package name and group name all match. src/main/uk/ac/soton/comp1206 directory structure with package uk.ac.soton.comp1206 the the top of App.java and specified correctly in the pom.
  • You cannot override the main or classpath of a jar at the command line. No -cp for you.
  • You cannot load jar files from a classpath inside a jar file. You have to include Class-Path: lib\file1.jar lib\file2.jar etc.
