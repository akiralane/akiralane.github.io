There are three types:

  • Conceptual → Ideas
  • Logical → High level design
  • Phyiscal → Low level

Conceptual

Directly from the requirements and domain; no thought to actual database design.

Includes entity names, relationships and sometimes attributes.

Logical Model

Identification of attributes and attribute types.

Physical Model

Consideration of database structure, including actual tables and fields.

Provides a concrete implementation of relationships using keys, indices, join tables etc.

Cardinality and Modality

These are the indicators of the business rules around a relationship.

Cardinality, as outlined above, refers to the maximum number of times an instance of an entity can be associated with instances in the related entity.

Modality refers to the minimum number of times an instance can be related to instances in the related entity.

Keys

Primary Key

A primary key ensures every row is unique.

You can't have two rows with the same value(s) for the primary key.

A primary key can be on a single field or multiple fields.

If you don't provide one, SQLite uses an automatically generated rowid

Foreign Keys

Foreign keys help ensure referential integrity between tables.

If a column in T1 references a column in T2, we can't have a value in T1 that isn't in T2.

Foreign keys are created when the table is created

FOREIGN KEY (column) REFERENCES Table(column)

Foreign key constraints have to be turned on for this to work.

The pragma command changes the way the DBMS is set up; we can use it to turn on foreign key support.

PRAGMA foreign_keys = ON;

Maintaining referential integrity

We can tailor this behaviour using ON DELETE and ON UPDATE.

FOREIGN KEY (column) REFERENCES Table(column) ON DELETE <action> ON UPDATE <action>

Potential actions:

CASCADE the deletion/updates to the referring tables.

Might not be always appropriate - if you delete a booking, you don't want to delete the person from your database!

RESTRICT prevents the parent being updated/deleted.

SET DEFAULT sets the key value to the default. SET NULL has similar behaviour, in the way you'd expect.

Surrogate Keys

Things like automatic ID fields are what we term surrogate keys; they have no business meaning.

Surrogate keys are attributes created and maintained in the context of the system to aid in uniquely identifying an instance of an entity - they do not have any relation to the real world.

They're used because natural keys aren't generally stable, consistent or efficient enough to use in a database.

They are added as part of physical design, after logical modelling and normalisation.

Don't overdo it though!
