Abstract Data Types

In the object-oriented methodology you separate the interface from the implementation. The actual implementation is encapsulated and may be changed without affecting how the class is used.

The interface is the set of public methods.

An abstract data type (ADT) specifies a set of operations through which a certain data structure can be accessed. You are "entering into an agreement" that you only intend to use the underlying structure in the declared way.

Common ADTs include stacks, queues, sets and maps. Each has a limited set of operations on it.

There are many possible implementations for a given ADT: for example, a stack can be implemented as an array which can only be accessed via pop, push and peek.

By declaring your intentions with an ADT, you are making your algorithms easier to understand and maintain.

Stacks

Stacks are LIFO memory - Last In First Out.

They can be accessed using:

  • Tpop(): remove the top item and return it,
  • push(item): add an item to the top,
  • Tpeek(): return the top item,
  • bool isEmpty(): true if stack is empty, false otherwise.

Stacks are useful because the interface is very simple. It reduces the access to memory, which prevents interactions in a way which may break the program.

They are sufficient for many applications: for example, matching parentheses or XML tags can be done with stacks quite intuitively.

Arithmetic expressions can be evaluated via stacks by pushing a number to the number stack, pushing the operator to an operator stack and then pushing the other number to the number stack. The stacks can now be reduced down using the information from both and this process can be repeated with following arithmetic elements.

Queues

Queues are FIFO memory - First In First Out.

They can be accessed using:

  • enqueue(item): add an item to the end of the queue,
  • T peek(): return the item at the front of the queue,
  • T dequeue(): removes and returns the item at the front of the queue,
  • bool isEmpty(): true if queue is empty, false otherwise.

Priority queues are also a thing; they hold the data each bound to a numerical representation of their priorities.

They have the methods:

  • insert(item, priority): add an item at priority priority,
  • T findMin(): return the item with the highest priority (smallest numerical value),
  • T deleteMin(): remove the item with the highest priority.

Queues and priority queues are used in operating systems for things like print/job queues, or communication/message passing. Priority queues are used specifically for greedy algorithms, like Prim's minimum spanning tree algorithm.

The most efficient implementation of a queue uses a heap, surprisingly.

Lists

A list is a collection where the order of elements is important.

I won't list the methods because you've used them so much.

Java has a List<T> interface and provides specific implementations for this interface like ArrayList<T> and LinkedList<T>.

A linked list is a collection of elements where each element points to the next - their order is not given by their physical placement in the collection. This allows for efficient insertion or removal of elements from any position in the sequence during iteration.

There's more detail on linked lists and arrays on the next page.

Sets

Sets model mathematical sets. There is no order or repetition in a set.

Operations: add, remove, contains, size, isEmpty. You can imagine what those do.

Java has a Set<T> interface which is commonly implemented using HashSets or TreeSets. The best one to use depends on the application.

Program to Interfaces, not Implementations

Use data structures through their interfaces:

Set<String> someSet = new HashSet<String>();
List<Integer> someList = new ArrayList<Integer>();

This makes modification and maintenance easier because it's not implementation dependent.

Declare your intentions, not your actions; program to interfaces, not implementations.
