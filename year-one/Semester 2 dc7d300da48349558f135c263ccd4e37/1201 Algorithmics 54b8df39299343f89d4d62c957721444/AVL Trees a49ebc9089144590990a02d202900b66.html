AVL Trees

An AVL tree is a binary search tree in which:

  1. the heights of the left and right subtree differ by 1 level at the most
  1. the left and right subtrees are themselves AVL trees

This guarantees that the worst case AVL tree has logarithmic depth.

Implementing AVL trees

To implement an AVL tree, we need to include additional information at each node indicating the "balance" of the subtrees.

balanceFactor = (height of left subtree) - (height of right subtree)

So if a tree is an AVL tree, then the following are the only options for the balanceFactor:

−1-1−1﻿ when the right subtree is deeper than the left

000﻿ when the subtrees are of equal depth

111﻿ when the left subtree is deeper than the right

We want to maintain this when implementing an AVL tree. This means we need to:

  1. update balanceFactor after each add or remove operation,
  1. rebalance the tree using rotations if the balanceFactor goes outside of the defined bounds

To summarise:

  1. Find the location where the element is to be inserted and insert it.
  1. Iterate up through the parents, adjusting the balanceFactor
  1. If the balance factor of a node exceeds ±1\pm1±1﻿ then re-balance the subtree of that node and stop.
  1. Else, if the balance factor of the node is zero then stop.

In Step 3, we can stop because the height of the affected subtree after the insertion will be the same as before once you rotate it. Hence, no node outside of this subtree will be unbalanced (assuming you were already working with an AVL tree).

Deletion

AVL deletions are similar to AVL insertions, in that you need to update the balance factors.

One difference is that after performing a single rotation, the tree might still not be within the AVL criteria so higher levels would need to be examined.

In the worst case, Θ(log⁡(n))\Theta(\log(n))Θ(log(n))﻿ rotations would be necessary. That is, every subtree on the path to the root from the deleted node would have to be rotated (since AVL trees have depth log⁡(n)\log(n)log(n)﻿.

This is slow, but in many applications deletions are rare.

AVL Tree Performance

Insertion, deletion and search in AVL trees are, at worst, Θ(log⁡(n))\Theta(\log(n))Θ(log(n))﻿.

The height of the tree is Θ(log⁡(n))\Theta(\log(n))Θ(log(n))﻿, so searching is Θ(log⁡(n))\Theta(\log(n))Θ(log(n))﻿ at worst.

Insertion without balancing is Θ(log⁡(n))\Theta(\log(n))Θ(log(n))﻿. Balancing takes an additional Θ(log⁡(n))\Theta(\log(n))Θ(log(n))﻿ steps in the worst case - but this is an addition, so it doesn't affect the rate of growth.

The above is the same for deletion.

This means that, in the worst case, AVL trees are considerably better than a basic binary search tree when searching.

However, insertion is slightly slower in an AVL because we often need to perform balancing as well.
