What to use...?

Our motivation for hashing is content-addressable memory - like a telephone directory, we can perform lookups according to the object's contents.

If the list is already sorted, we could use Binary Search to search in Θ(log⁡(n))\Theta(\log(n))Θ(log(n))﻿. A binary search tree would obviously be the best solution for this.

We can do better than that...

Indexing

Let's call the content we want to search on the "key". For a telephone directory, the key would be the name of the person we want to call.

We can achieve O(1)O(1)O(1)﻿ search, insertion and deletion if we use the key as the index for a big array. If we made the key a string which could be represented as an mmm﻿-digit binary number, we could just look up the key in a table of 2m2^m2m﻿ items.

How we do it

We could naively only label each memory address in our space and place data somewhere in there.

This wastes a bunch of space, however.

We can save on memory by "folding" the table into itself using modulo. Check out Computerphile's video on the maths behind Diffie-Hellman for a really nice visualisation of how using modulo does this!

Hashing functions

A hashing function x.hashCode() takes an object x and returns a positive integer - this is x's hash code.

To turn the hash code into an address, take (hash code % the table size):

int index = Math.abs(x.hashCode() % tableSize)

If tableSize =2n=2^n=2n﻿, we can compute this more efficiently using a bitmask(?).

int index = Math.abs(x.hashCode() & (tablesize - 1))

What makes a good hashing function?

We want similar objects to be mapped to different integers. We also want to avoid mapping two different objects to the same hash, as much as possible (this is known as hash collision).

The function should have certain qualities:

  • fast to compute
  • keys should be distributed as evenly as possible to avoid collisions
  • consistency with the equality testing function - equal objects should have identical hash codes

A good general-purpose hash function is

hash(k)=k%Mhash(k)=k\%Mhash(k)=k%M﻿

where MMM﻿ is a prime number used as the table size. The fact it is a prime number greatly reduces the chance of collisions - again, check out the Computerphile video on Diffie-Hellman's maths.

Collision Resolution

Collisions are inevitable and need to be dealt with. Two commonly used strategies for collision resolution are:

  • Separate chaining: Make a hash table of lists - each cell of the hash table points to a linked list of records which have that hashcode. This uses less memory but can have a time cost, as we see later. (The function used in the example below is k%7k\%7k%7﻿)
  • Open addressing: Find a new position in the hash table. That is, we just keep moving down the table until we find an empty slot (known as a linear probe). This means that each entry's location does not necessarily directly correspond to its hash. Searching an openly addressed table is as simple as going to where the element should be and iterating down until we find the entry we want.

    Open addressing uses more memory due to the need to re-size the table, but can be faster.

    However, the entries will tend to pile up (known as clustering), since multiple entries will have the same hash and will be added in the next free space. Clusters become worse as the number of entries grow.

    The proportion of full entries in the table is known as the loading factor. More on this and clustering later.

Resizing an openly addressed table is pretty easy: create a new hash table of some larger size, iterate through the old hash table, add each element to the new hash table.

You'd have to re-compute all the hash codes, obviously. However, the amortised cost of this entire process is small.

The size of a hash table is a classic example of a space-time tradeoff, since using sparser hash tables improves execution time - especially in open addressing, because you won't have to move down the table as much.

Searching a separately chained hash table

To find an entry in such a hash table, we again use the hash function on a key to find the table entry, then search the linked list at that entry.

The time complexity depends on the objects hashed:

  • If the objects are evenly dispersed in the table, search and insertion is O(1)O(1)O(1)﻿.
  • If multiple objects are hashed to the same entry in the table, then search becomes O(n)O(n)O(n)﻿, since you have to search the linked lists at the entries as well.

To iterate through a separately chained hash table, we simply iterate through the array of entries and, for each hash, we iterate through the linked list at that point.

This becomes more efficient as the table fills up, since we're doing less iterating over null entries (where there is no linked list present).

Reducing the number of probes in open addressing

Recall that naively moving down the table to find a desired cell (empty for insertion or non-empty for search) is a linear probe. Also recall that the proportion of full entries in the table to the total size is known as the loading factor λ\lambdaλ﻿.

The number of probes needed for insertion and searching increases with the loading factor λ\lambdaλ﻿. With linear probing, this is made worse due to the tendency to cluster.

With λ=0.9\lambda=0.9λ=0.9﻿ (1 in 10 locations are free),

  • without clustering, the expected number of probes is 101010﻿
  • linear probing would typically require around 505050﻿ probes for insertion

To avoid clustering, two other strategies are commonly employed: quadratic probing and double hashing.

Quadratic probing

In quadratic probing, we look for the i2i^2i2﻿th slot in the iii﻿th iteration if we collide. For the iii﻿th collision, we try locations:

hash(x)+i2hash(x)+i^2hash(x)+i2﻿

That is, we take steps 0,4,9,16,…0,4,9,16,\dots0,4,9,16,…﻿

Quadratic probing prevents primary clustering so it dramatically decreases the number of probes needed to find a free location when the table is reasonably full.

One problem is that if we're unlucky, we might not be able to add an element to the hash table even if the table isn't full.

The problems with removal in open addressing

If we remove an element, we potentially create a gap in the "run" of elements which would be checked by a probe. This means that removing one element could inadvertently prevent the discovery of many other elements.

Lazy Remove

One easy fix is to mark the deleted cell with a special dummy entry.

The find method would consider such an entry to be full and an iterator would ignore the entry. However, an insert operator would be able to insert a new entry in this location.

What to use...?

Most libraries use separate chaining. Java included.

The performance does not degrade badly as the number of entries increases and it also reduces or removes the need to resize the hash table.
