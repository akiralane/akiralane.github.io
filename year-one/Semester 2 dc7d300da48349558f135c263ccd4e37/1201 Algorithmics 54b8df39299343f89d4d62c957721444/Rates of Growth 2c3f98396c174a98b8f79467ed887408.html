Rates of Growth

Remember that the rate of growth being referred to here is the rate of growth of the time complexity T(n)T(n)T(n)﻿, as nnn﻿ gets larger.

Big Theta

Suppose we have an algorithm that has a worst-case number of elementary operations given by 4n2+12n+1994n^2+12n+1994n2+12n+199﻿.

We're interested in the asymptotic behaviour: what happens when nnn﻿ becomes very large.

For this, we take the leading term 4n24n^24n2﻿ because 12n+19912n+19912n+199﻿ becomes irrelevant for large nnn﻿. Also, we take the functional behaviour - ignore the constant 444﻿.

So, this algorithm would be called an order n2n^2n2﻿ (or quadratic time) algorithm.

This is written as Θ(n2).\Theta(n^2).Θ(n2).﻿

Advantages:

Captures the idea that two functions have the same rate of growth. If f(n)f(n)f(n)﻿ is Θ(g(n))\Theta(g(n))Θ(g(n))﻿ then f(n)≈cg(n)f(n)\approx cg(n)f(n)≈cg(n)﻿ for big nnn﻿ and some constant c>0c>0c>0﻿.

We can estimate algorithm run times on large inputs by measuring run time on a small input. If I have a Θ(n2)\Theta(n^2)Θ(n2)﻿ average-case algorithm which I've measured to take xxx﻿ seconds on an input size 100100100﻿, then it will take about x×n21002x\times n^2\over100^21002x×n2​﻿seconds on an input of size nnn﻿.

Disadvantages:

Can't compare algorithms with the same time complexities.

Can be misleading for small inputs, because the trailing terms are discarded and they may make a difference. However, we rarely ever care about small inputs. What's the point of an algorithm then?

Counting Operations

The time complexity of an algorithm is often easy to calculate.

You can count the number of nested loops to find the order.

Bounds

When we have conditionals, time complexity is harder to compute. It no longer only depends on the input size, but also the evaluation of the conditional statement!

For example, if there's a loop inside an if statement inside a loop, the order will be somewhere between nnn﻿ and n2n^2n2﻿ because the nested loop may or may not happen.

To avoid having to think really hard, we can look for upper and lower bounds for the rate of growth of time complexity.

These do not change between worst-case and average-case.

If we take the previous running time function 4n2+12n+1994n^2+12n+1994n2+12n+199﻿, then:

it executes no more than order n2n^2n2﻿ operations. It is O(n2)O(n^2)O(n2)﻿.

it executes no fewer than order nnn﻿ operations. It is Ω(n)\Omega(n)Ω(n)﻿.

These do not have to be tight bounds. If an equation is O(n)O(n)O(n)﻿ then it's also every OOO﻿ asymptotically bigger than that. You're just saying that it executes no more than that many.

Use and Misuse

  • Big-O notation is most common
  • People often say they have O(n2)O(n^2)O(n2)﻿ when they have a Θ(n2)\Theta(n^2)Θ(n2)﻿, which is a stronger statement because O(n2)O(n^2)O(n2)﻿ is not a tight upper bound.
  • An O(n2)O(n^2)O(n2)﻿ algorithm is also an O(n3)O(n^3)O(n3)﻿ algorithm. See quoted text above.
  • An O(n2)O(n^2)O(n2)﻿ may not be faster than a O(n3)O(n^3)O(n3)﻿ algorithm, even for large nnn﻿.
  • An Θ(n2)\Theta(n^2)Θ(n2)﻿ algorithm is always faster than an Θ(n3)\Theta(n^3)Θ(n3)﻿ algorithm for large nnn﻿. Notice that this is for Θ\ThetaΘ﻿, not OOO﻿.
