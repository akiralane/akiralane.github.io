Event-B

Event-B has its foundations in the basis of set theory - operations like union, intersection, disjoint etc. are all present.

The type of a set of something is the powerset of the set of the thing's type. I bet that's going to be easy to read when you need it!

Building Example

Here's an example which models a system which tracks if users are in a building or not...?

The Context defines information about the environment in which the machine operates.

  • Carrier Sets: abstract types used in specification
  • Constants: as you'd expect; variables whose value does not change
  • Axioms: constraints on the constants. An axiom is a logical predicate. See the invariants below for a potential analogy...?

The Machine defines rules about how the system should act.

  • Sees: the contexts in which the machine operates. I don't know why BuildingContext isn't declared here...
  • Variables: state variables whose values can change, obviously.
  • Invariants: constraints on the above variables. An invariant is a logical predicate.
  • Initialisation: the inital values for variables which need them
  • Events: "guarded" actions specifying ways in which the variables can change. Events can take parameters.
    • Here, "guarded" means that a condition must be fulfilled for the event to trigger.

Here's an example of an event which adds a new user:

  1. take any element which is in the set USER but not in the set register - that is, all unregistered users,
  1. then, union the set register with the singleton of the given user and assign this new set to register. Effectively, add the user element to the set.
  1. to preserve invariant 2, all users must be inside or outside. I think out was picked for no particular reason here, but also it would make less sense to add a user to the system when they're already inside - people should be registered before coming into the building, probably.
