When Big-O doesn't work

The underlying assumption of Big-O is that all elementary operations take about the same amount of time. This simply isn't true of disk lookup:

  • The typical time to carry out an elementary operation on a modern processor is 10−910^{-9}10−9﻿ seconds.
  • However, a typical hard disk might do about 120 revolutions per second...
  • The typical time it takes to locate a record is around 10ms - this is 10710^{7}107﻿ times slower than an elementary operation!

So obviously, we want to minimise the number of disk accesses. This is critical for good performance when a program is reading from memory - for example, in database applications.

Storing data in binary trees is disastrous here. We typically need log⁡2(n)\log_2(n)log2​(n)﻿ disk accesses before we locate our data. It's not unusual in databases for n=107n=10^7n=107﻿, so log⁡2(n)≈24\log_2(n)\approx 24log2​(n)≈24﻿. This sucks - it would take several seconds to find a single record.

M-way trees

To remedy this, we can use M-way trees (trees where each node can have up to M children) so that the access time becomes log⁡M(n)\log_M(n)logM​(n)﻿.

(This comes from log2(n)log⁡2(M){log_2(n)}\over\log_2(M)log2​(M)log2​(n)​﻿)

In practice, we might use a 200-way tree so that we can reduce the factor by around a factor of 8 (200≈28)(200\approx 2^8)(200≈28)﻿.

The basic structure for this is a B-tree...

B-Trees

Not to be confused with binary trees!

B-trees are a type of balanced trees designed for fast search, insert and delete. The way they are set up keeps related data close to each other in memory to minimise retreival time - this is especially important when working with large amounts of data which is kept on slower secondary storage.

Hence, B-Trees are used extensively in databases.

Rules

A basic implementation would obey the following rules:

  1. The data items are stored in leaves! The rest of the nodes just exist to point us toward the data.
  1. The non-leaf nodes store up to M-1 keys to guide the search: key iii﻿ represents the smallest key in subtree i+1i+1i+1﻿
  1. The root is either a leaf or has between 222﻿ and MMM﻿ children
  1. All non-leaf nodes except the root have between ⌈M/2⌉\lceil M/2\rceil⌈M/2⌉﻿ and MMM﻿ children
  1. All leaves except the root are at the same depth and have between ⌈L/2⌉\lceil L/2\rceil⌈L/2⌉﻿ and LLL﻿ data entries

The choice of MMM﻿ and LLL﻿ here depend on the block size (information read in one go from disk). It also depends on the data type.

Example

MMM﻿ and LLL﻿ might often be in the hundreds or thousands, but in the example here we consider M=L=5M=L=5M=L=5﻿ just so that's easy to visualise.

Search

Say we wanted to find 49. We see in the root node that our value is between 41 and 66, so we go down that path to the corresponding node. We see there that our value is between 48 and 51, so we go down that path to the block which contains our ⌈L/2⌉\lceil L/2\rceil⌈L/2⌉﻿ to LLL﻿ (here, 5) data entries which we can quickly search.

Insert

Insertion works in much the same way. We follow the path to where our value should be and insert it there. However, what if we wanted to insert 40? Following the path, it should go in the leaf to the right of 35 - but we see that this already has 5 entries in! It's full!!!!!!!!

When this happens, we simply split the parent node up into two new blocks of MMM﻿, split the leaf into two new blocks of LLL﻿ at the (ceiling) halfway point and change the parent node to point to our new leaf correctly by adding a key. This gives us a bunch of extra space.

Other changes

  • If the root is full then it can be split into two and a new root created.
  • B-trees also ahve to allow the removal of records without losing their structure.
  • A number of strategies exists to improve performance. No, I'm not going to tell you them.
  • The actual implementation of B-trees is tricky because there are many special cases.

Tries

"Trie" singular is pronounced "try" because the person who named it made it homophonous with "tree" and everyone decided that that was dumb as hell.

Tries (also known as digital trees??) are multiway trees often used for storing large sets of words. Their name comes from the word retrievalretrievalretrieval﻿.

All words in a trie have the special character $\$$﻿ which signifies the end of the word.

Uses

Tries are yet another way of implementing sets. They provide quick insertion, deletion and search. They are typically considerably faster than binary trees and hash tables where applicable.

Tries are particularly good for spell checkers, completion algorithms, longest-prefix matching and hyphenation(?).

Disadvantages

Tries usually waste huge amounts of memory, as you can see above. Often, tries are used for the first few layers, while lower levels use a less memory intensive data structure.

These days, memory is cheaper and less of a problem so tries are acceptable for some applications.

Suffix Trees

A suffic tree is a trie of all suffixes of a string. They're very important for string-base algorithms, like finding a match for a string in some text.

For example, to find a match we can construct the suffix tree of the text and then query the string we want in said trie. Trie is such a stupid name.

Using a regular trie for a suffix tree would typically use far too much memory to be useful. However, by using pointers to the original text, it's possible to build a suffix tree with O(n)O(n)O(n)﻿ memory usage where nnn﻿ is the length of text.

There also exists a linear time algorithm (O(n)O(n)O(n)﻿) to construct the trie from the text. This one's not trivial to understand, unfortunately.
