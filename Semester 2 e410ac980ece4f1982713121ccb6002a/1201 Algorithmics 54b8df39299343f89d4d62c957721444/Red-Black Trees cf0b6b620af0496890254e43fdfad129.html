Red-Black Trees

Red-black trees are another strategy for balancing trees, where nodes are assigned either "red" or "black" according to two rules:

Red Rule: the direct children of a red node must be black. You can't have a red node point to a red node.

Black Rule: the number of black nodes in all paths to a node with one or fewer children must be the same.

This ensures that no path from the root to a node with one or fewer children is more than twice as long as another, which means that the tree will be kept full.

Restructuring

When inserting a new element, we first find its position. If it's the root, we colour it black. Otherwise, we colour it red. This often means we need to restructure the tree.

How we do this depends on the new node's "uncle" - the sibling of the parent node.

Both Java's Collection and C++'s STL use red-black trees.

TreeSet

Java's Collection class has a TreeSet class implemented using a red-black tree. This iterates over the elements in order (unlike the HashSet). These are the classes to use if you want a collection of objects with no repetitions.

A TreeSet has a guaranteed order, unlike most sets. They are ordered using their natural ordering, or by a Comparator.

Maps

One major ADT we haven't seen an implementation for yet is the map.

The Java class TreeMap<K, V> implements maps using red-black trees.

We can implement Map using a TreeSet by making each node hold an Entry<K, V> object.

Conclusions for binary search trees in general

  • All types of BST are very efficient (order log⁡(n)\log(n)log(n)﻿ insertion, deletion and search) providing that they are balanced
  • Balanced trees are achieved by performing rotations
  • There are different strategies for when to rotate, including AVL and red-black.
  • BSTs are used for implementing sets and maps.
