Normalisation

We want to avoid redundant data.

We also want to avoid anomalies; these include

  • Update anomalies
    • Inconsistency, partial updates, having to update in mutliple places
  • Insert anomalies
    • Attributes can't be inserted without requiring the presence of other attributes
  • Deletion anomalies
    • Deleting a certain row might remove all references to a certain unique attribute that that row had (deleting a race entry which was the only inhabitant of a planet would remove any information you had about the planet, because that race was the only row which had data about the planet)

The steps to normalisation

We have several forms of hierarchical normalisation. We achieve these by decomposing our tables into smaller, more specific tables.

Each form also maintains the attributes of the previous forms.

  • First Normal Form
    • Contains only atomic values in attributes (no arrays etc.) and has no repeating groups (the same attribute across multiple columns; i.e. you can't get around "only atomic values" by adding more columns of the same attribute with different values).
  • Second Normal Form
    • Removes partial dependencies: non-key attributes must depend on every part of the primary key.
  • Third Normal Form
    • Removes transitive dependencies: no non-key attributes depend on another non-key attribute.
  • Boyce-Codd Normal Form (sometimes known as 3.5 Normal Form)
    • Every attribute must be a fact about a key: every determinant is a candidate key (can uniquely identify any row).

A table is in 3NF but not BCNF if:

The table has two or more candidate keys

At least two of the candadate keys are composed of more than one attribute

The keys are not disjoint: the composite candidate keys share some attributes

Primary key

To start, we need to find a primary key.

To find that, we need to find the candidate keys. Recall that they are sets of one or more attributes which uniquely identify a row.

A candidate key is always a determinant (on the left side of a functional dependency) because knowing it allows us to uniquely know other things; however, they are a special case in that they determine all attributes of a relation.

Each candidate key can qualify as a primary key - we select the best from the options.

Usually, that's the one which requires the least information to uniquely identify each row.

The domain

We need to consider the domain as well. What do we know about the context of our database that might not be present in the data?

Not all candidate keys for an existing set of data will hold for all possible entries into the database. For example, a new entry could break a functional dependency (a ruler conquering a second planet would break the dependency "ruler→planet" because you can no longer know the planet when given the ruler).

Composite keys

Sometimes, often to avoid issues with the domain, you will need a composite key - made up of multiple attributes - to uniquely identify.
