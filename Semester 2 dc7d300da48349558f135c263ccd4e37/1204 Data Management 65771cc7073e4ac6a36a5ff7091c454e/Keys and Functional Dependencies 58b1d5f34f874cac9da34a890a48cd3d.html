Keys

A set of attributes forms a key for a relation if we do not allow two different tuples in a relation instance to have the same values in all attributes of the key.

So, a key is a set of attributes which have a unique combination for each "entry". Consider the following relation:

Movies(title, year, length, genre, studioName, starName)

For a relation schema RRR﻿ and set of attributes XXX﻿ of RRR﻿:

  • A superkey is a set of columns that form a unique reference. XXX﻿ is a superkey of RRR﻿ if X→AiX\rarr A_iX→Ai​﻿ for every attribute AiA_iAi​﻿ of RRR﻿.
  • A candidate key is a superkey which has the smallest number of columns possible included. XXX﻿ is a key or candidate key of RR R﻿ if XXX﻿ is a minimal superkey of RRR﻿. That is, if XXX﻿ is a superkey of RRR﻿, there is no superkey Y⊂XY \subset XY⊂X﻿.

{title, year, starName, length, studioName} is a superkey.

{title, year starName} is a candidate key.

Functional Dependencies

Let a relation with schema R(A1,…,An,B1,…,Bm)R(A_1,\dots,A_n,B_1,\dots,B_m)R(A1​,…,An​,B1​,…,Bm​)﻿ and let rrr﻿ be an instance of RRR﻿.

We say that rrr﻿ satisfies the functional dependency

A1,…,An→B1,…,BmA_1,\dots ,A_n \rarr B_1,\dots ,B_mA1​,…,An​→B1​,…,Bm​﻿

if whenever two tuples in rrr﻿ agree on the values of A1,…,AnA_1, \dots , A_nA1​,…,An​﻿ then they also agree on B1,…,BmB_1, \dots, B_mB1​,…,Bm​﻿.

In other words, there are no two tuples in rrr﻿ that share the same values on the attributes A1,…,AnA_1, \dots , A_nA1​,…,An​﻿ but differ on B1,…,BmB_1, \dots, B_mB1​,…,Bm​﻿.

In this case, we say that A1,…,AnA_1, \dots , A_nA1​,…,An​﻿ functionally determines B1,…,BmB_1, \dots, B_mB1​,…,Bm​﻿.

The set of "AAA﻿" attributes is the determinant and the set of "BBB﻿" attributes is the dependent set. We can just write this as A→BA \rarr BA→B﻿ if they are both well defined under those names.

Rules for Functional Dependencies

Splitting/Combining rule

If A1,…,An→B1,…,BmA_1,\dots ,A_n \rarr B_1,\dots ,B_mA1​,…,An​→B1​,…,Bm​﻿ then

A1,…,An→B1A_1,\dots ,A_n \rarr B_1A1​,…,An​→B1​﻿

A1,…,An→B2A_1,\dots ,A_n \rarr B_2A1​,…,An​→B2​﻿

⋯\dotsm⋯﻿

A1,…,An→BmA_1,\dots ,A_n \rarr B_mA1​,…,An​→Bm​﻿

Implication

An FD A→BA\rarr BA→B﻿ is logically implied by a set SSS﻿

(denoted S⊨A→BS \models A\rarr BS⊨A→B﻿)

if every relation instance that satisfies all the FDs in SSS﻿ also satisfies A→BA\rarr BA→B﻿.

Equivalence

SSS﻿ is equivalent to TTT﻿ if and only if S⊨TS\models TS⊨T﻿ and T⊨ST\models ST⊨S﻿.

That is, SSS﻿ and TTT﻿ are satisfied by exactly the same relation instances.

A relation schema RRR﻿ satisfies a functional dependency if every instance rrr﻿ of RRR﻿ satisfies it.

How do we determined whether or not a set of attributes is a superkey of a schema? How do we find all the superkeys of a relation?

Alternatively, from Matthew Barnes:

  1. Start with the set XXX﻿. This is X0X_0X0​﻿.
  1. Look along FFF﻿ and check if there are any Y→ZY\rarr ZY→Z﻿ where Y⊆XY\sube XY⊆X﻿.
  1. If there is, then union X0X_0X0​﻿ and ZZZ﻿ and make it X1X_1X1​﻿. We now proceed using X1X_1X1​﻿.
  1. Keep repeating around FFF﻿ until XnX_nXn​﻿ remains the same each iteration.
  1. This should give you the closure X+X^+X+﻿.

This seems a little overcomplicated, but it lets us find superkeys and identify candidate keys computationally.
