UML Class Diagrams

Object modelling models the actual clases and entities used in the system.

Class diagrams partition the system into areas of responsibility (classes) and describes the associations between them.

They can be at three distinct levels:

  • Conceptual: shows concepts in project domain
  • Specification: shows interfaces between components
  • Implementation: shows classes which correspond directly to code

Classes

A class is represented as a box split in three: name, attributes, methods in that order.

These qualities are assigned symbols which describe their scope:

  • +: public
  • #: protected
  • ~: default
  • -: private

Multiplicity works the same way as use case diagrams.

You can also label associations, for clarity.

A class can be associated with itself. An example is that employees manage other employees; managers are themselves employees.

A common mistake is to represent actions as if they were associations! Class diagrams should show how classes relate to each other - the actual operations should be methods of the class. If you're very concerned about actions, use a different UML diagram!

It's not good to have many duplicate attributes. If a subset of a class' atributes form a coherent group, then it could be better to create a distinct class containing these attributes.

Association classes

Association classes describe properties which don't fit in one class or another, since the properties represent an association between those classes.

  • Let’s say you have students and they’re studying courses. That’s simple; a course can have many students, and a student can study many courses.
  • But now you want to store what semester they’re in, what optional modules they’ve picked etc.
  • Do we store these attributes in Student or Course? Neither, because they’re attributes that describe the association, not the actual classes.

Both the above syntaxes are valid.

Generalisation

Generalisations in class diagrams are analogous to inheritance in OOP (sometimes it will directly describe inheritance, if you are drawing a class diagram at the implementation level).

The label on a generalisation, sometimes called the "discriminator", describes the criteria used in the specialisation.

Aggregation and Composition

Aggregations are special types of associations meaning that one class is "part of" another class. They are represented by a hollow diamond at the end of the association line. Composition is a strong kind of aggregation. They are represnted by a filled diamond at the end of the association line. In the below example, there are rooms in a building. If you were to destroy the building, you'd have a hard time finding any rooms in it. You know, since there's no building there anymore.

Alternatively, a chessboard has 64 squares. If you got rid of the chessboard, the squares would necessarily disappear as well.

Object diagrams

An object diagram is a class diagram which specifically describes the instances of the classes.

Names of objects are in the form [object name]:[class name], such as jim:Employee or coolSquad:Company.
