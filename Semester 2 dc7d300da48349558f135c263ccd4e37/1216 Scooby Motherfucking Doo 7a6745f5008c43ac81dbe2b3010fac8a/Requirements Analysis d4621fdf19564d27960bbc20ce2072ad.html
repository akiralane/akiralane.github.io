Types of Requirements

  • Functional
    • What does the program actually do for the user? Focus on "what", not "how".
      • Normal: regular functionality
      • Error: error handling
      • Safety: safety of the user themselves - lift shouldn't move while doors are open
      • Security: safety of the data and the system - login needs a username and password
    • Example: "It must allow the user to login"
  • Nonfunctional
    • Qualities of the program. Unnecessary for functioning, but typically very desirable. Still a requirement.
    • Examples: "It must be multi-platform", "it should not take longer than 10 seconds to load"
  • Design
    • Defines how the system will operate. It's debatable whether or not design concerns are technically requirements.
    • Avoid specifying these in the requirements phase, since they may overly constrain the system when it is time to implement it.
    • Example: "It will use a MySQL database to store user details"
  • Constraints
    • A global requirement which is put in place out of necessity.
    • Example: "Must be able to function on 1GB of RAM"
  • Assumptions
    • Properties of the environment that are required for the system to function properly. Outside of the program's control.
    • Example: "User passwords are kept secret by the user"

Defining Scope

The scope provides a high-level view of the planned system in terms of:

  • Need: what need is it intended to meet?
    • "Our website sucks; we need a better website"
  • Goals: what functionality should it provide? What is the intended purpose of the functionality?
    • "To allow customers to access our services more quickly and conveniently"
  • Case: what is the business case for the product? Why is this product good for the business?
    • "Allows the business to stay competitive by increasing internet presence and making interacting with the business easier for customers"
  • Stakeholders: actors who interact with and are interacted with by the system. This includes customers, administrators, maintainers, developers, testers, third party services etc.
    • "Customers, marketing dept., ordering and dispatch dept., accounts dept., IT dept., web hosting company(?)"
  • High-level operational concepts: a high-level view of how the product will be used; typically as a list of actions which can be carried out by the stakeholders.
    • "Customers can use keywords to find items via search"
    • "Customers can see product information and related products"
    • "Customers can add items to their basket, checkout the basket and pay"
    • "Customers can track their orders"
  • Assumptions: properties of the environment that are required to be considered for the system to function properly; outside of the system's control
    • "Customers will interact with our website on both desktop and mobile devices"
  • Constraints: global requirements which are put in place out of necessity
    • "Complete within 6 months on a budget of £150k"

Operational Scenarios and Use Cases

A use case is a list of events or actions which abstractly describe the interactions between the user and the system. It should be general and cover almost every possibility. It has multiple components:

  • Name, Actors, Entry Condition, Flow of Events, Exit Conditions, Exceptions, Quality Requirements

A scenario, however, is a documentation of a forseeable series of interactions between the user roles and the system. Apparently it can be considered as an instance of a use case.

Verification and Validation

Verification is an equivalence check between the transformation of two models.

Validation is the comparison of the model with reality - requirements should be validated with the client and the user.

Validation involves several checks:

Correctness - represents client's view

Completeness - all scenarios well-defined

Consistency - no contradictions

Concreteness - lack of ambiguity in the system; the model describes one system, not several

Realism - model can be implemented
