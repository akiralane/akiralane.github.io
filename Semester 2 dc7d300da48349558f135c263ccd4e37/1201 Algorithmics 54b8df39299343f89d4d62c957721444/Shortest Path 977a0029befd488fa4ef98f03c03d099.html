Shortest Path

Given a connected, undirected, weighted graph, we want to find a shortest path from a source sss﻿ to all other nodes.

Idea:

Each node uuu﻿ maintains its distance d[u]d[u]d[u]﻿ from sss﻿ (initialised to be infinity).

When we examine a new node vvv﻿, update the distance of vvv﻿'s unchosen neighbours if going through vvv﻿ is shorter than their current distance.

Dijkstra's Algorithm

Input: A graph G=(V,E)G=(V,E)G=(V,E)﻿, edge weights wew_ewe​﻿ and a root vertex sss﻿.

Output: the shortest distance d[u]d[u]d[u]﻿ from uuu﻿ to sss﻿.

To my understanding, the effective idea of Dijkstra's is to go down the path of least resistance until it becomes larger than some other path we could have taken.

Example, using a directed graph

The only difference with using a directed graph is that the adjacent vertices are changed.

In the above graph, Dijkstra's would visit the nodes in order S, C, A, D, F, E, B.

Here's a step through:

S → C is the shortest path we can take from S to any other immediate node. However, any edge we take from there would produce a path greater than...

S → A is the next shortest, with a "resistance" of 3.

A → D is the next shortest with a length of 4, and does not make the path length greater than any other path we've come across so far. However, the only edge from D is D→E, which would add another 4 and make it greater than...

S → F. Moving to E from here would give us a path length of 8, but then again, so would...

A → B is the last to be connected as it has a path length of 9, which is larger than any other path we could take. This completes the tree.

Pseudocode

So we start with an array of nodes. We initalise the distance from the start to each node as infinity as a placeholder for "we don't know". We also have another array for "previous", which will hold the neighbour node that you would need to pick for the shortest path, working backwards. We also have another array for the nodes we've visited. (this is the easiest implementation to understand in my opinion, but there's probably one which takes less memory).

We then check all the vertices from our starting node and update their distances.

Then, we pick the one with the shortest distance. We set its "previous" to the node we just came from.

We check the vertices of this new node as well (and give the nodes they point to distances), but remember that we are checking the distance from the start node, so we also want to check it against all other non-visited non-infinite nodes in the graph.

When we find the smallest, we update its "previous" value to the node that came before it (not necessarily the node we were checking!) and continue from there.

This repeats until all nodes are visited.

Then, to find the shortest path, we can trace backwards through our "previous" from the node we want to go to towards the start node.
