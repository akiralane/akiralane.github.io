Quicksort partitioning

Recall that in quicksort, we partition our array and recursively quicksort the partitions. We want a good method for partitioning.

Lomuto Scheme

In the Lomuto scheme, we "scan" the whole array with an index variable jjjÔªø and keep track of our partition with an index variable iiiÔªø.

Whenever we find an element A[j]A[j]A[j]Ôªø which is less than the pivot's value, we swap the elements at iiiÔªø and jjjÔªø and increment iiiÔªø.

This means that iiiÔªø always marks the division between the elements less than the pivot and the numbers greater than or equal to the pivot. Remember that the pivot can be anywhere in the array, even for the majority of jjjÔªø's traversal - although it will always end up in the correct place once the partition is over.

// using the rightmost element as the pivot
// remember that left and right contain INDICES, not the values
procedure PARTITION_LOMUTO(arr, left, right)
	pivot := arr[arr.length-1]        // the rightmost element is always the pivot in this algorithm
	i := left - 1
	for (j := left) to (right - 1) do // j traverses the array
		if arr[j] <= p                  // if this element is less than the pivot
			swap(arr[i], arr[j])          //    swap this element with the element at i
			i++                           //    increment i, marking more elements as less than the pivot
	swap(arr[i], arr[right])          // finally, swap the rightmost element (the pivot) with the element at i
	return i+1                        // return the new index of the pivot

Since this is a linear search through an array, the Lomuto scheme runs in O(n)O(n)O(n)Ôªø.

This method is relatively easy to understand, but we can do better...

procedure PARTITION_HOARE(arr, pivot, left, right)
	i := left - 1                    // i starts at the left
	j := right + 1                   // j starts at the right

Quicksort improvements

Shuffling

We'd like to avoid worst case performance, where our partitions are small/huge respectively.

Randomly shuffling our input decreases the chance of picking a bad pivot.

Knuth Shuffle

  1. Loop backwards through the array
  1. At every iteration, generate a random index in the unshuffled part of the array.
  1. Swap that item with the current element.

This has complexity Œò(n)\Theta(n)Œò(n)Ôªø, obviously, and is performed in-place.

Multi-pivoting

A modification of quicksort which uses two pivots results in improved efficiency. This is the default implementation now used in Java.

This doesn't change the number of comparisons or swaps. The reason behind the better performance is largely due to fewer cache misses in real systems.

The details are beyond the scope of this module.

Assumptions in radix sort:

  • We have an array of size nnnÔªø, as is standard.
  • Each item in the array is an exactly dddÔªø-digit word or number.
  • Each digit can take kkkÔªø possible values - kkkÔªø is the base, or radix.

Basic procedure: for each digit moving from dddÔªø to 1, sort the entire array by that digit using a stable sort.

procedure RADIX_SORT(arr, d)
	for (i := d) to (1) do
		use a stable sort to sort arr by digit i

Think of this like how you'd sort a bunch of 3-letter words lexicographically: you sort words by their last letter, then the second, then the first. Since the first letter is the most significant in the ordering, it is considered last.

procedure COUNTING_SORT(arrIn, arrOut, k)
	let counterArr := new array [0,...,k] filled with zeroes
	
	for (i := 1) to (arrIn.length) do
		counterArr[ arrIn[i] ] += 1               // count distinct items in input array

		counterArr[i] += counterArr[i-1]          // cumulatively count

	for (i := arrIn.length) to (1) do
		arrOut[ counterArr[arrIn[i]] ] = arrIn[i]
		counterArr[ arrIn[i] ] -= 1

step 1 > go through the list, count how many of each value there is and store in an array step 2> make the array into a cumulative count array step 3> go through the original array backwards, placing at the position given in the cumulative count array
