Multithreading

Quick recap: threads run in "parallel", waiting for the scheduler to execute their instructions (not code line by code line! It's per JVM instruction).

In Java, you construct some sort of Runnable:

public class stuffHappenObject implements Runnable { 
	public void run() { ... }
}

and then put it into a thread and run it:

var threadA = new Thread(new stuffHappenObject());
var threadB = new Thread(new stuffHappenObject());

threadA.start(); // start, NOT run. run will simply call the run method in the Runnable in the current thread
threadB.start(); // start will run the run method in the new thread.

States of a thread

  • new ‚Üí freshly allocated
  • runnable ‚Üí running - that is, available for the scheduler to start queueing its instructions
  • blocked ‚Üí waiting for a lock, which I will cover in a moment
  • waiting ‚Üí waiting for another thread
  • timed waiting ‚Üí waiting for a fixed amount of time, as in the Thread.sleep() method
  • terminated ‚Üí finished, available for garbage collection

Useful methods

  • setName(String name), getName(String name) ‚Üí name usually gets set in the constructor, but it can be useful to rename it. Getting a thread's name can be useful also.
  • isAlive() ‚Üí true if the thread has been started
  • join() ‚Üí pauses execution of the calling thread until the thread that this is called on finishes. That is, don't continue until this is done - "join" the threads.
  • setDaemon() ‚Üí makes the thread a daemon thread.
    • daemons are low-priority threads which run in the background. Their purpose is to provide services to user threads
    • the JVM will terminate if the only running threads are daemons
    • threads created within a daemon will themselves be daemons
    • a daemon thread must be marked as such BEFORE it is started

A race condition is the event that multiple threads compete to modify/access the same public resource, which can lead to unexpected behaviour.

To avoid this, we use synchronisation. That is, we lock a resource so that no other threads can access the resource while our thread deals with it.

Every object in Java has a "mutual exclusion lock" or mutex associated with it. This means that we can tell the JVM that only the object with this mutex can access the associated code while under synchronisation.

In Java, there are two ways:

// code block
synchronized (obj) {...}

This presents a code block which gets passed the object to be used as the lock. All code in the brackets will be synchronised under the object.

// method signature keyword
synchronized void method() {...}

This means that all code in the method is synchronised when called. The default value for the mutex here is this - the object which the method belongs to.

You can use synchronisation in a static context by using ClassName.class as the lock.

We can also use multiple objects as locks, if we want to control access to a resource more specifically. (e.g. "A makes sure B isn't accessing the resource, but B makes sure A isn't accessing the resource")

synchronized (a) {
	synchronized (b) {
		...
	}
}
