Types and Functions

Haskell is statically typed. The compiler knows your types at compile time and doesn't let you write type conflicts. Types can't change during runtime.

Types are written like this, read 'e has type t':

e :: t

Every expression has a type, often calculated during runtime using type inference.

Basic and Compound types

There are two flavours of type: basic and compound. These are about what you'd expect: basic types include old friends like Bool, Int, Char etc.

Compound types are formed with type operators. They include list, tuple, function etc. For example, [Bool], (Int, Char), Bool -> Int.

For each compound type there is a data constructor (typically just called a constructor) which builds the values.

The constructor for a tuple is ( , ), as you'd expect.

Tuples can be made from multiple types at a time, unlike lists.

The constructor for a list is called "cons"? It's a colon : and (:) :: a -> [a] -> [a].

As you can see, it's a little circular - you need to give it a list in order to make a list. Thankfully, we can always use the empty list [] to start a list off.

In general, lists are built as sequences of applications of the cons operator:

1 : 2 : 3 : [] gives us [1, 2, 3].

The second of the above notations is also valid as a list constructor - it's simply syntactic sugar for the chained cons.

Functions

Note how I said earlier that 'function' is a compound type. Functions have a single parameter type and a single return type.

So, we can use the fact that functions can be built out of other types to make functions which return functions! These are called curried functions.

In fact, all functions in Haskell are curried â€” all functions in Haskell take just a single argument.

The type notation's arrows are right associative, while function application is left associative, so the types fit.

add' :: Int -> Int -> Int -- is actually Int -> (Int -> Int)
add' x y = x + y
-- here, we're actually writing (add' x) y

Here, add' is a function of type Int -> Int. That is, it returns a function Int -> Int.

So, add' 3 gives us a function which adds 3 to some parameter, so (add' 3) 2 yields 5.

This is how we obtain functions with multiple arguments in Haskell - or rather, the appearance of multiple arguments.

Partial application

This view of functions lets us partially apply functions.

Have a look at this function:

multThree :: (Num a) => a -> a -> a -> a -- is a -> (a -> (a -> a))
multThree x y z = x * y * z

What happens when we do multThree 3 5 9?

As we know from the left associativity of function application, this is actually ((multThree 3) 5 ) 9.

First, 3 is applied to multThree. That creates a function which takes a parameter and returns a function which returns a function that multiplies its parameter by 3. 5 is then applied to that, which creates a function that will take a parameter and multiply it by 15 (since the innermost function was evaluated or something?). 9 is applied to that new function, which gives us 135.

That's a lot to get through, at least for notes. Take a look at this:

multThree x y z = x * y * z

multTwoThingsWithNine = multThree 9
-- (multTwoThingsWithNine 2 3) returns 54
-- see how (multThree x) returns returns its own function? 

multWithEighteen = multTwoThingsWithNine 2
-- (multWithEighteen 10) returns 180

By calling functions with 'too few parameters', we're creating new functions which we can still use!

We can feed some values to a function, and leave others for later on.

compareWithHundred x = compare 100 x -- we don't really need to supply both values to compare yet...
compareWithHundred = compare 100     
-- so here we partially apply compare, and we can fill in the value to compare
-- with later!
