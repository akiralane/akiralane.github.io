Writing Functions: Pattern Matching

When we can know the cases of our function based on the input's shape rather than its value, we can use pattern matching. (We can also pattern match on value, but it's usually far less concise than using guards.)

You'll notice a lot of the same structure as Rust's match.

Here's an example:

nonZero :: Int -> Bool
nonZero 0 = False
nonZero _ = False

Notice the wildcard _ matching all other cases.

Here's some examples of the destructor pattern, where we unpack compound types using pattern matching with the constructor of those types:

fst :: (a, b) -> a
fst (x, _) = x

snd :: (a, b) -> b
snd (_, y) = y

See how we don't have to bind values that we don't use to a name - the wildcard can be used repeatedly, unlike actual names.

For example, (_, x, True, _) matches any 4-tuple where the 3rd value is True, binding the 2nd value to x and discarding the 1st and 4th values.

List patterns

Similar to how we can use the tuple constructor to match tuples, we can use the list constructor : to build patters which match lists.

This is what we've been doing the whole time with the (x:xs) pattern!! The list constructor takes an element on the left and a list on the right - hence, this pattern matches a single element x followed by the rest of the list xs.

Here's an example:

last :: [a] -> a
last [] = error "can't take the last of an empty list!"
last (x:[]) = x
last (x:xs) = last xs

It doesn't have to stop there though. (x:y:z:xs) only matches a list with at least 3 values and binds them to the corresponding names.

Remember that strings are just Char lists. We can apply any of this list pattern matching to strings too!

Composite patterns

Where pattern matching gets really powerful is where you combine patterns between multiple arguments. You often have to rely on the top-down evaluation, though.

fetch :: Int -> [a] -> a
fetch _ []     = error "can't fetch from an empty list"
fetch 0 (x:_)  = x
fetch n (_:xs) = fetch (n-1) xs

When in doubt, list each combination of case individually. This can help with readability too.

When to use patterns or guards

It comes down to structure.

Use patterns when the structure can identify the definition cases.

Use guards when only the values identify the definition cases.
