This is the one where we map programs to mathematical domains.

The first step here is obviously then to identify the semantic domain to which we’re going to map programs.

Elements in the semantic domain represent the meaning of programs.

  • For programs which return positive integers, a reasonable choice of semantic domain is the naturals.
  • For programs which represent functions from integers to integers, we could choose the set of all functions between naturals.
  • For programs which return pairs of integers, we could take the semantic domain to be the Cartesian product of the naturals with itself.
  • etc.

Typically, semantic domains are built by following the structure of the types of the language.

In an ideal language, the structures of the types would make for well-known, simple mathematical structures in the domain!

This isn’t always the case, though - effectful functions, loops and recursion end up complicating things.

Comments on denotational semantics

It doesn’t give a very clear account of how a program is actually supposed to execute.

Instead, they give a precise and nicely compositional account of which values the program is supposed to calculate - it abstracts away all of the execution steps.

This can be useful for modelling pure functional languages, but it can be tricker for modelling languages with mutable state or concurrency, for example.

Recursion is also tricky for denotational semantics.

What value does a non-terminating recursive loop get mapped to?

A criticism of the above denotational model of the Toy language is that there’s a lot of “junk” in the model - the semantic domain [[Int→Int]][[\text{Int}\rarr\text{Int}]][[Int→Int]]﻿ is all functions from N\NN﻿ to N\NN﻿. This includes uncomputable functions - it’s too big, in a sense.

There’s a lot of research into finding denotational models which are “just right”. This is hard, even for small toy languages!
