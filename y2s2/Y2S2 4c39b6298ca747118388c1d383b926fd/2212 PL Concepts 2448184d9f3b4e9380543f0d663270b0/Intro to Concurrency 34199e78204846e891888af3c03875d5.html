Intro to Concurrency

Parallel computing is distributed and independent. For concurrency, the threads are interacting.

For threads to interact, they must be able to synchronise.

Fundamentally, this means that threads have to signal to each other - this forms the basis for all inter-thread communication. The method, however, varies greatly between systems.

A couple of paradigms

Signal and wait:

  • A thread that signals another thread sends the signal and then enters a waiting state so that it can’t be scheduled further until the signal is received.
  • Synchronous.

Signal and continue:

  • A thread that signals to other threads doesn’t need to wait until the signal is received.
  • Asynchronous. Sending threads don’t need to rendezvous, but obviously received ones still have to wait.

Semantics-first

You might consider encoding it as a relation which looks like this:

(T1,T2,…,Tn)→(T1′,T2′,…,Tn′)(T_1, T_2, \dots, T_n) \rarr (T_1', T_2', \dots, T_n')(T1​,T2​,…,Tn​)→(T1′​,T2′​,…,Tn′​)﻿

This isn’t really concurrency, though - there’s no interaction between threads. It’s just parallel computing.

The semantics of individual threads can’t be considered in isolation.

Each thread must consider the environment in which it executes - memory, other threads, interrupts, etc.

With this in mind, we can write small-step semantics which look like this:

TundefinedactionT′T \xrightarrow{action}T'Taction​T′﻿

Here, action can be effectful - it can include both threads’ interactions and computational steps internal to the thread.

A calculus for concurrency - CCS

A particularly well-know calculus for concurrency is called CCS (Calculus of Communicating Systems). It’s one among many examples of what’s called a process calculus.

Here’s the grammar:

P ::= nil | a!P | a?P | τP | P||P | P\a | X | rec X . P

Where:

  • nil is an inert process - it has terminated.
  • a!P sends a signal on a channel called a and then continues execution as P.
  • a?P receives a signal on a channel a and continues execution as P.
  • τP takes an internal computation step.
  • P||P represents two concurrent processes - also models thread spawning.
  • P\a is a restriction operation; no communcation on channel a originating inside of P can be seen.
  • X and rec X . P are used for recursive behaviour.
    • For example, rec X . a! b? X is a process that repeatedly sends a signal on a and receives a signal on b.

An example CCS process

The symbols between the terms are ||, representing concurrency.

Server thread P1 publishes an invitation to client threads P2 and P3 to respond on channel b. The first to do so gets to interact with P1 while the other is blocked.

Client thread P2 or P3 calls channel s some number of times and calls z to finish. Process P4 counts the number of signals sent on channel e.

The semantics

The semantics of CCS are typically presented using synchronous communication. It’s straightforward to consider an asynchronous version, though.

We simply have to adjust the grammar to forbid any continuation of the process after a send signal - then the only way to continue computation is on a new thread.

Non-determinism

Note that there’s no determined behaviour - it depends on which threads get to which channels first.
