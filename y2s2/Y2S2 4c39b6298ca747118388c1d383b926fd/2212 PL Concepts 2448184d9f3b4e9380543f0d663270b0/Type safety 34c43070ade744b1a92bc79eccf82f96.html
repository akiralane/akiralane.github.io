So in that slogan “well-typed programs never go wrong”, what do we mean by “wrong”?

Notice that the operational semantics we gave for the Toy language didn’t depend on the terms being well-typed:

if(if(true) then false else 0) then true else 34 evaluates to 34 despite being ill-typed.

if(if(false) then false else 0) then true else 34 gets stuck.

There is no rule to reduce if(0) then true else 34.

Stuck terms and divergence

In the small step semantics of a language, we can model the runtime errors as “stuck” terms. That is, any terms which can’t evaluate any further and are not values of the language.

For the big step semantics, we could do the same - but this means (if we added recursion or looping) that we couldn’t distinguish between stuck and divergent terms.

To model runtime errors in a big step semantics we have to explicitly model errors. This would be done by introducing an error relation that describes when EEE﻿ has reached an error state.

Type safety for big step

The big step semantics only considers ‘complete’ computations - it can’t refer to what happens during computation.

All that we can easily specify with big step semantics is a weak notion of type correctness: preservation.

Progress

Preservation alone isn’t a sufficient property to describe type safety.

It doesn’t capture the fact that well-typed terms stay in semantically ‘good’ states.

For example, if we have a (broken) type system that allows a stuck term to be well-typed, then this stuck term will trivially satisfy preservation.

Remember that “stuck terms” are effectively error states - they are terms of the language for which there are no reduction rules, like if (5) then _ else _. A good type system will prevent these.
