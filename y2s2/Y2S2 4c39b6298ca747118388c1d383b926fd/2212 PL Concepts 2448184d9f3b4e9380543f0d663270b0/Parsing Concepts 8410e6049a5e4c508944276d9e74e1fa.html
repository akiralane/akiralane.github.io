Parsing is the process of converting a stream of tokens into an AST.

This stream of tokens is typically generated via lexing.

There are two common approaches to parsing:

  • top down or recursive descent
    • This builds the parse tree from the root downwards.
    • Associated with LL grammars (left-to-right, leftmost derivations)
  • bottom up
    • This builds the parse tree from the leaves upwards.
    • Associated with LR parsers (left-to-right, rightmost derivations)
    • More complicated, but handles a wider range of grammars.

LR parsing

Shift and reduce

The most common approach to implementing LR parsers is to use shift-reduce parsing.

The token stream is scanned and a forest of partial parsing subtrees is built up.

At each stage, the parser can either:

  • Shift: read the next input token, adding a new single node tree to the forest.
  • Reduce: apply a completed grammar rule and join subtrees together into a larger subtree.

This continues until the input is fully consumed and a single full parse tree is formed.

This kind of parser acts like a state machine, with actions ERROR, SHIFT, REDUCE, STOP.

Conflicts occur when the state machine is non-deterministic. Weâ€™ll come back to this later.

Shift-reduce conflicts

Shift-reduce conflicts arise from ambiguous parsing.

For example, when matching 1 + 2 + 3 after reading 1 + 2, the parser could either:

  • Shift and read the next token +
  • Reduce and complete the production for 1 + 2 into its own expression.

This is non-deterministic and therefore causes a conflict.

You can avoid this by structuring your grammar differently - layer non-terminals to imply associativity.

Another way to remove shift-reduce conflicts is to specify operator precedence using directives.
