Consider the expression:

The function just needs a record with a field labelled x, but the argument also has a field labelled y.

Another (smaller) example is that a function which returns the length of an array doesn’t need to care about the type of the elements in the array.

But type rules as we’ve defined them so far require us to specify the types precisely.

We’re looking for some sort of generic - or polymorphic - type.

Three common routes include parametric polymorphism, subtype polymorphism and ad-hoc polymorphism. We’ll be looking at subtype polymorphism.

Subtype polymorphism

We could think of types as structured sets and simply say that T<:UT<:UT<:U﻿ iff [[T]]⊆[[U]][[T]]\sube[[U]][[T]]⊆[[U]]﻿ (where the square brackets are their representations as sets).

This is a nice general definition, but it doesn’t give us a convenient syntactic description.

We have two other common options:

Nominal subtyping

We could just explicitly declare what types we want type be subtypes of others and then make sure that any operations valid on a supertype are valid on the subtype.

This is the approach taken in object oriented languages, through inheritance.

Structural subtyping

We could look at the capabilities of the types and say that TTT﻿ is a subtype of UUU﻿ if every operation that can be performed on UUU﻿ can also be performed on TTT﻿.

This definition incorporates lots of structural properties of the types - for example, pairs must be subtypes of pairs because of the projection operations.

Nominal subtyping

Under nominal subtyping, types are distinguished by their names alone - even if they represent the same structure.

For example, in:

struct Foo = {x: int, y: int}
struct Bar = {x: int, y: int}

A function which expects Foo can’t accept Bar, even though they’re structurally identical.

Java overcomes this by only allowing subtyping between a single form of structuring - classes.

It can then simply force that every member in the supertype also exists in the subtype.

This approach is typical of nominal type systems.

Structural record subtyping

Record types are a generalisation of pair types, and the subtype relation on records generalises in an interesting way too.

Record types don’t rely on syntactic positioning for indexing - only the labels matter.

This means we can write a record with some fields missing and have it be perfectly well formed as a record value of another type.

We couldn’t do the same with a tuple.

The rules for subtyping on records can be built up in stages.

First, we have the generalisation of what we saw just now for pairs. This is called depth subtyping.

That is, for two records that have the same number of fields, each of the members in the record can subtype the members in the other.

Then, we have the notion of width subtyping. This is where there may be extra fields in the subtype.

Finally, we allow re-ordering of the listed fields, since syntactic order doesn’t matter to us.

Not all languages follow these principles. Java, for example, doesn’t allow depth subtyping.
