Data comes in various shapes and sizes. To check that our programs are doing what we want them to, we can at least specify their behaviour in terms of the shape of the data that they manipulate.

This is where structured types come in.

The goal is to catch common programming errors where data being passed around is in a different format to what something was expecting.

The unit type

This is a surprisingly useful type.

It has exactly one value, written (). We’re going to write it as unit, for clarity.

This is the input type of Java methods which take no parameters, for example.

It can be used to suspend the evaluation of an expression until a later point in the computation.

For example, compare the following:

let x = print "Hello" in ... x ...
let x = fun () -> print "Hello" in ... x() ...

In a CBV language, the former immediately prints “hello” and continues evaluation whereas the latter wraps the print statement in a function for later use.

This operation of wrapping an expression with a function of unit type is called thunking. We can also unthunk a thunked value (presumably by evaluating the suspended computation).

Constructors and destructors

We call the form (E1,E2,…,En)(E_1,E_2,\dots,E_n)(E1​,E2​,…,En​)﻿ a constructor.

You can see why - it constructs a tuple.

There are corresponding operations that we refer to as destructors.

For tuples, these are called projections.

You’ve already come across the projections for pairs - they’re fst and snd.

We can also use pattern matching to take apart the structured data:

match E with (E1, E2) -> ...

Of course, for generalised nnn﻿-tuples, we need nnn﻿ projection functions.

Record types

We can generalise tuples even further - they are essentially collections of nnn﻿ elements of data which are indexed somehow.

We can use labels instead of integers to index the items. This is what’s known as a record!

You’ve seen records before, as structs in C and objects in Java (that is, every object is a record type!).

The usual syntax for record constructors is {I1=E1,I2=E2,…In=En}\{I_1=E_1,I_2=E_2,\dots I_n=E_n\}{I1​=E1​,I2​=E2​,…In​=En​}﻿ where III﻿ represents the labels (or fields) and EEE﻿ represents the values.

The type of the data of this shape looks very similar:

{I1:T1,I2:T2,…In:Tn}\{I_1:T_1,I_2:T_2,\dots I_n:T_n\}{I1​:T1​,I2​:T2​,…In​:Tn​}﻿

The destructor for this type is called selection and has a very familiar notation - we write record.field to select the field labelled field from the record record.

Non-structured types

Arrays

Arrays, on the other hand, aren’t really a structured type despite feeling similar. They have no constructor to form elements of the type and we can’t pattern match across them.

This is because arrays aren’t built up using a constructor - they are allocated all at once.

Functions

This isn’t structured because you don’t build functions up from a constructor.

Intuitively, it doesn’t make sense to ask for “all functions which match this pattern”, since every function consists of the form \x -> e - there is no structure to change.
