Types are useful in considering correctness of code - they can be considered a type of “lightweight” formal method.

They’re abstract and precise, which makes them useful for reasoning about a program’s properties without having to analyse the implementation.

Types of types

  • Strongly typed
    • Languages check that types match and fail if they don’t.
  • Weakly typed
    • Languages don’t always prevent mismatching types, instead using types for other compilation purposes (like checking the shape of data when allocating memory for a variable. C is weakly typed).
  • Static typing
    • Typechecking occurs at compile time
  • Dynamic typing
    • Typechecking occurs at runtime

What does it take to prove type safety?

“Well-typed programs never go wrong” - Robin Milner

In precise terms, what does that actually mean? What do we need to know for this?

  • “Well-typed”
    • We give an inductively defined typing relation. This looks a lot like a proof in propositional logic.
  • “programs”
    • We give an inductively defined reduction relation. This describes how the program runs in abstract terms.
  • “never go wrong”
    • We give a description of the error states of the program and use the mathematical descriptions above to try prove that type safety holds for a given language.

How do we know that code is well-typed?

For static typing, the compiler needs to check that, at any point in a program’s execution, the program doesn’t attempt to consume data of the wrong type.

Certain operators consume data, so the compiler needs to find where these operators get used.

This amounts to checks of the AST according to types.

Typing relations

As just mentioned, every program fragment E needs to be given a type T in order to build up a picture of whether the whole AST is well-typed.

We write this as a typing relation, ⊢E:T\vdash E:T⊢E:T﻿ (read “E has type T”).

We need to defined this relation for every possible program E.

We will want to do local checking on syntax trees - this means that the type of a program should only depend on the types of its children.

Note that only certain operators generate actual checks for correct usage of types.

At some points, we may need to “approximate” the type where we can’t determine it statically.
