We identify the virtual entities that we manipulate in programs using names, or identifiers.

Variables

Originally, a variable referred to a memory location whose contents could change - i.e., a reference.

Now, it‚Äôs more generalised to mean ‚Äúa placeholder for some possibly complex value‚Äù. This is a semantic concept, not just some part of memory (except when it is, like C).

‚ÄúAliasing‚Äù refers to two variables pointing to the same memory location.

Usually, you don‚Äôt want this - it can lead to unexpected behaviour. However, some algorithms and data structures make use of it.

Attributes

A variable typically has six attributes we associate with it:

  • Name
  • Address (L-Value, the left-hand side of an assignment)
  • Value (R-Value, the right-hand side)
  • Type
  • Extent - the time between its allocation and deallocation. Also known as lifetime.
  • Scope

Bindings

A binding is an association between an entity (usually a variable) and some attribute.

e.g. between a variable and its type, or a variable and its scope.

Some entities don‚Äôt need to be bound at runtime while others do - it‚Äôs useful then to distinguish between static and dynamic binding.

As you know, static bindings happen at compile time and remain unchanged throughout execution while dynamic bindings occur (and can change) during runtime.

Pretty much every attribute (apart from name?) can be bound either dynamically or statically.

Allocation and deallocation

We refer to the binding of a variable to its address attribute as allocation.

This can be either static or dynamic, while deallocation is pretty much just a dynamic concept.

As noted above, a variable‚Äôs extent is the time between its allocation and deallocation.

Four kinds of variables

  • Static (global)
    • These are allocated at initialisation time.
  • Stack-dynamic (local)
    • Memory is allocated from a runtime stack and bound when a declaration is executed. It gets deallocated when the block containing it returns.
  • Explicit heap-dynamic
    • Abstract memory locations which are explicitly allocated by the programmer.
    • This includes things like malloc and free in C, new and delete in C++, new() in Java.
  • Implicit heap-dynamic
    • Memory in the heap is allocated when variables are assigned to values and deallocated/reallocated with re-assignment.
    • This is error-prone and inefficient. Used in ALGOL, LISP, C and JavaScript (for arrays).

Types

Static type binding is usually either done through explicit type declaration or through type inference.

Type declaration is most commonly used. At the point where a variable is introduced, you tell the compiler its type at the same time.

Type inference doesn‚Äôt require type annotation of variable declarations - the compiler works it out from how the variable gets used.

There‚Äôs a shit version (which is arguably just explicit declaration) where types are ‚Äúinferred‚Äù by how the variables are named. In Perl, $p is a number or string, @p an array etc.

Hindley-Milner inference (introduced by the ML language) deduces the most general type for a variable through its usage. The most general type usually gets expressed using polymorphism or generics

Dynamic type binding sucks, mostly.

JavaScript, Python, Ruby, Lua all do it.

This can have benefits for convenience and readability, but there are obvious downsides.

Extent

For the third time on this page, extent is the period of execution for which a variable is bound to a particular location storing a meaningful value.

This is a semantic concept - what extent means, precisely, depends on the execution model.

A running program can enter and leave a given extent multiple times - like in the case of a closure.

The four kinds of variables have different extent:

Static variables have an extent of the entire program duration.

Stack-dyamic variables have an extent of a particular stack frame or procedure.

Explicit heap-dynamic variables have an extent from explicit allocation to explicit deallocation - it‚Äôs all managed by the programmer. This can cause memory leaks.

Implict heap-dynamic variables have an extent from implicit allocation to implicit deallocation. Values may persist in memory despite their addresses being freed.

Scope

The scope of a variable is the part of the code in which it can be referenced.

That is, it‚Äôs the part of the program where its name is meaningful.

A variable‚Äôs scope affects its extent; a value which can no longer be referenced is pretty much meaningless. Garbage collectors exist to clean these up.

For local variables, the scope is usually the block it‚Äôs declared in.

For static variables, the scope is usually the entire program except where they get hidden by locally scoped variables of the same name.

We refer to lexical scope where scope is aligned to statically determined areas of code, like between the two parentheses of a block or a method body.

Dynamic scope

You can have dynamic scoping, but it‚Äôs uncommon and kinda sucks. Perl and Lisp use it.

As the name implies, scope is determined at runtime, reliant on program flow. You can have one method call another and variables from the first method stay in scope even in the second.
