Embedded systems are typically in harsh environments and also can be used in critical systems. This is a bad combo!

There’s also a legal dimension - embedded software is typically part of a product and liability can’t be dodged by excluding it in the license.

Common issues

Hardware

  • Glitches:
    • Power
    • Clock
    • Signal degradation (interference, capacitance between wires etc.)
  • Wear-out:
    • Flash
    • EEPROM

Software

  • Timing
    • Real-time deadlines
    • Race conditions
  • Difficult code
    • Parallellism / concurrency
    • Optimisation

The best we can do is to reduce the probability that something bad happens.

Don’t underestimate the failure probabilities for a mass-market product which is on 24-7!

Mitigation measures

  • Physical protection
    • Radiation-hardened hardware
  • Targeting application-specific failure modes
    • Make sure you’re failing in the right way. If you’re in an airplane and you detect an error, you don’t want to just shut everything down.
  • Redundancy, as written about in the previous set of notes.
  • Protection circuits
    • Includes things like a watchdog timer
    • See below for more.
  • Defensive programming
    • Things like filling free memory with “jump to reset” instruction, for example.
    • If you do fail, it shouldn’t be destructive.
    • See below for more.

How should we respond to a failure?

Shut down?

Usually requires some sort of redundant system.

Try to recover?

Use a reset like a watchdog?

Is recovery within real-time requirements?

Elaborating on “protection circuits”

Use things like radiation or temperature sensors to recognise adverse situations and prepare.

Recognise failure:

Watchdogs, or checksums.

Use fallback systems when you deviation too much from expected behaviour.

This is a last resort!

Elaborating on “defensive programming”

Use self-stabilising protocols:

Random data in message streams should be dissipated out of the system after being passed around enough.

Initialise unused resources:

For example, the thing where you write a jump instruction to a reset vector to all unused memory.

Good coding practice:

Use MISRA C, for example.

Use automated checkers.
