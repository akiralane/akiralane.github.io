Operating systems

Operating systems, at their core, provide several basic functions.

They provide a uniform interface - for both users and programs.

They manage resources. This includes things like CPU time, memory and access to devices.

They manage interactions. These can be desired interactions (networking, communication between processes) or undesired (regarding protection and security).

Multiprogramming

This is where multiple programs are available for execution on a single CPU. It’s pseudo-parallelism.

If one process needs to wait for IO, then the CPU can work on another process.

Processes

A process has an entry in the process table which usually contains:

  • Instruction counter, stack pointer
  • Assigned address space (where the OS will intervene if the process leaves this)
  • Associated resources
  • State

Process States

Running: currently using the CPU. There can only be one of these at a time, since we’re not talking about real concurrency.

Ready: runnable, but stopped to let another process run.

Blocked: unable to run until an external event occurs.

As you can see, to be “preempted” is to be stopped to let another process run. Processes typically alternate bursts of computing with IO requests.

The operating system needs hardware support for interrupts to provide this.

Scheduling

The requirements for the scheduler differ significantly according to the nature of the processes and the system.

Some considerations: fairness, response time, throughput, turnaround, CPU utilisation, deadlines (hard/soft), predictability, adherence to policy.

Scheduling algorithms

  • First-come first-served
  • Shortest job first
    • Not always possible to tell! Halting problem etc etc.
  • Round robin (requires pre-emption and interrupts).
    • Time slices - known as quanta - are assigned to each process in equal portions in circular order.
    • This means it’s perfectly fair (and simple!), but there’s no notion of priority.
      • Short quantum means high context-switching overhead, but shorter response time.
      • Long quantum means better efficiency (likely to block for IO before quantum expires) but longer response time.
  • Shortest remaining time
    • Same problem as shortest job (but allows for more accurate estimates?)
  • Priority scheduling
  • Lottery scheduling
    • Where each process gets a “fraction” of the CPU based on their priority.
      • Tickets are chosen at random.
    • Fundamentally no guarantees on time → bad for real-time systems.

Context switching

If we have a preemptive OS, the OS can switch between processes.

Interrupts facilitate this.

Sometimes, we want to switch between user and kernel processes.

There must be a hardware divide between the two - otherwise the distinction isn’t really meaningful.

Real-time scheduling

A real-time scheduler is under more constraints that a normal one. It has to be:

  1. Correct (obviously)
  1. within a deadline
    1. could be soft - like multimedia
    1. could be hard - like flight control systems
  1. within a power budget
In a hard real-time system, a late result is no better than a stupid idiot garbage incorrect one.
