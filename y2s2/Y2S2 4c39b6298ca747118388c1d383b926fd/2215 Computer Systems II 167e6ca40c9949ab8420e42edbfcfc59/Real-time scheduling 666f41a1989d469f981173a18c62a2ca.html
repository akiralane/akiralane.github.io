Assumptions for analysis

How do we schedule it?

We need predictable worst-case performance.

Simple case

If all tasks can be finished within the period of the most frequent task, we can use static cyclic scheduling.

We just use a table to assign slots to tasks at compile time. This isn’t really scheduling, but it’s the ideal situation.

Fixed priority scheduling

This is scheduling where the priority of the tasks is decided at compile time.

We like this, because it’s simple and well-understood.

We have confidence in the behaviour, and code is relatively easy to verify.

Rate Monotonic Scheduling

Requirements for RMS:

The tasks are independent. There is no blocking each other.

The tasks have fixed CPU requirement.

Context switching is free (again, not a problem)

The deadline is the task period.

What if your task-set doesn’t meet that bound?

Are we wasting 31% of the CPU?

We can run non-real time tasks as low priority in the remaining CPU time.

We can also optimise the task periods to achieve a regular execution pattern. This is what we’re about to see in harmonic task sets.

Harmonic task sets

A harmonic task set is where every task’s period is a multiple of the period of any higher priority task’s.

If possible, make your tasks harmonic and schedule them with RMS.

RMS can reach 100% utilisation with harmonic task sets!

Gaps between tasks can accumulate, so getting rid of them really improves things.

This also means you can impose tighter deadlines and run task sets that you couldn’t before!

Harmonic tasks sets are also easy to analyse - they have a regular execution pattern.

Dynamic priority scheduling

If dynamic priority is required, we use Earliest Deadline First. It does what it says on the tin - the deadline which is closest gets highest priority.

This can have utilisation up to 100%, with and without preemption.

This does incur more overhead and more complexity in the scheduler.

Although, benefits include changing importance of tasks, accommodating new tasks during runtime and variable execution times.

It’s not stable under overload, though.
