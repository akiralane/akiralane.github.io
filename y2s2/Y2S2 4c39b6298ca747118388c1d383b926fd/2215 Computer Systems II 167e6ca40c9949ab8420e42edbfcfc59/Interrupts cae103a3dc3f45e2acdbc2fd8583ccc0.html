Basic IO methods

Programmed IO - or polling - is where the CPU sits in a tight loop until input is available or output can be accepted.

This occupies the CPU, but facilitates very fast reactions.

Alternatively, we can use interrupt-driven IO. This is where a hardware signal changes the program flow. The CPU can do other work (or sleep) until IO is possible.

Other IO methods like DMA or dedicated IO computers (like graphics cards) use interrupts to communicate with the CPU.

Interrupts

The CPU needs to be able to continue where it was interrupted after the interrupt has been dealt with.

A “precise interrupt” must meet certain conditions:

  • The state of the program counter (PC) is known.
  • Everything before the PC has been fully executed.
  • Nothing after the PC has been started.
  • Execution state of the instruction at the PC is known (Process Status Word).

What happens when an interrupt arrives?

The processor completes the current instruction and the acknowledges the interrupt.

The hardware saves some state, like the PC and the process status word.

The PC is then loaded with a value from the interrupt vector table - the control is now handed to software.

The software disables interrupts and saves additional required state (like putting registers on the stack).

At this point, the software can reenable interrupts, but it’s easier to think about if it doesn’t.

Then, it services the interrupt by jumping to the Interrupt Service Routine (ISR), at the position loaded from the interrupt vector table.

After that’s done (hopefully as fast as possible), it restores the saved state.

The hardware can then restore the PSW and PC.

Rules for ISRs

  1. Keep them fast. If they’re fast, you can usually just block all other interrupts while the ISR runs, which makes our life much simpler.
    1. Avoid loops - makes it hard to place bounds on running time
    1. Avoid heavy instructions (printf)
    1. Avoid instructions which block (scanf)
  1. Keep them simple
    1. Debugging ISRs is hard, since they aren’t part of normal program flow.

Latency

Latency is the time from the moment the interrupt arrives to the moment the CPU starts servicing the interrupt.

Making latency deterministic can be important in real-time systems.

Both algorithms and human operators are really quite good at adapting to deterministic latency, but random delays are harder to deal with.

We are often typically interested in maximum latency. We like putting bounds on things!

Causes of latency

Hardware

  • Waiting for current instruction to complete. This is technically non-deterministic but the variance is often very low.
  • Time taken for hardware support to save program state.

Software

  • Software also needs to save state.
  • The (maximum) length of critical sections - where we’ve disabled interrupts. This is a large cause of non-deterministic latency.

Jitter

What’s the variance in our latency?

Some instructions take multiple clock cycles, although this is quite low (on AVR there are only two types of instruction: one-cycle and two-cycle).

How do we keep ISRs fast and simple?

So, we want to minimise the length of the critical sections, but we also want to keep interrupts off during an ISR (effectively making the entire run time of the ISR a critical section).

Keeping interrupts off means that we have no worries about stack depth arising from “nested” interrupts, and there’s no overhead from re-entrant code.

However, this does mean a potential increase in latency and lost interrupts (if we’re not servicing them fast enough).

Then, what do we do? One trick for an ISR:

  1. Move the data that needs processing into some buffer.
  1. Set a global flag. volatile
  1. Return immediately.

Then, the main loop can continually check the flag and process the data if it’s there.

Hence, the time-critical bit has been done in a time-critical manner - the rest can be done eventually.

ISRs in C

C itself has no support for ISRs. It’s compiler-specific!

For example, the Pi Pico SDK has

irq_set_exclusive_handler(uint num, irq_handler_t handler).

Each IRQ has a macro with its own number. For example, PIO0_IRQ_0 is 7.

Note that C statements are often compiled into multiple instructions - this means that statements can be interrupted mid-statement.
