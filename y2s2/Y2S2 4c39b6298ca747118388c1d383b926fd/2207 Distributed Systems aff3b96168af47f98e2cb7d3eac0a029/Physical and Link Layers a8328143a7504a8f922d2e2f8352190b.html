Layer encapsulation

Each layer adds its own header. This becomes part of the payload for the layer below it.

The link layer

The purpose of the link layer is to shield upper layers from the specifics of the connection type.

There are a wide range of physical media. We need standards for transmission of bits over each type so that the link layer can use them.

For example, IEEE802.11 standards define the transmission for WiFi (which, yes, is part of the physical layer).

Other PHYS media include coaxial cable, twister pair, powerline, fibre optic, wireless, laser, sound etc. Some of these are more common than others, obviously.

The link layer transmits frames over physical media as well as receiving them, passing IP datagrams back up the stack.

Additionally, it detects and handles transmission errors.

Variation

The link layer can vary from end-to-end. Each section might have its own physical and link layer.

Flow control

The link layer can also regulate flow of data, so that a slow receiver is not swamped by a fast sender.

Link layer acknowledgements (ACKs)

There are three general link layer models:

  • Connectionless, no acknowledgements
    • This is for very low error rate networks, like wired Ethernet.
    • It’s “connectionless” in the sense that no path is established in advance - the frames get sent and may or may not get to their destination.
  • Acknowledged, connectionless
    • This includes wireless 802.11 - WiFi.
    • 802.11n supports block acknowledgements, where you can send a bunch of data at once and it’ll tell you which ones it didn’t get.
  • Acknowledged, connection-oriented.
    • This is for long-delay, unreliable links, like satellite.

Handling ACKs and errors

Obviously, there are a range of strategies available.

The simplest is stop-and-wait Automatic Repeat reQuest (ARQ).

You send a frame, wait for the ACK, send next frame, etc.

You can improve ARQ with pipelining:

Send multiple frames before receiving the first ACK.

Go-back-n ARQ: label each frame in flight - if ACK gets missed, retransmit from there.

Selective-repeat ARQ: similar to the above, but only retransmit the frames you lost.

Error detection and correction

Very few connections are error free. Radio certainly isn’t.

The simplest method for detecting errors is using a parity bit.

This isn’t robust enough, though. One example of a better solution is Cyclic Redundancy Check - CRC.

Both the sender and receiver calculate the checksum and compare.

Checksums can happen at higher layers as well.

MAC - media access protocol

A MAC protocol manages access to and from the PHYS layer. It is part of the link layer.

It has a mechanism for sending frames to and from PHYS and typically manages channels, frequencies and collisions. This is where MAC addresses come in.

Link layer example: WiFi

WiFi is essentially the wireless alternative to Ethernet.

Devices associate with a wireless access point.

The medium is shared (radio) so there can be significant contention in busy areas. We need some sort of collision avoidance scheme.

Handling media contention

The original wired Ethernet used Carrier Sense Multiple Access with Collision Detection. That’s a long name. (CSMA/CD)

Effectively, the sender just listens to see if the medium is busy. If it is, the sender waits.

Back off before retransmitting if a collision gets detected.

WiFi doesn’t use CSMA/CD because WiFi devices generally can’t send and receive at the same time.

There’s also the “hidden node” problem - devices see the Wireless Access Point but not each other.

Instead, WiFi uses CSMA/CA (which is CSMA/Collsion Avoidance).

CSMA/CA is like CD but instead of listening to the transmission, it waits for an acknowledgement from the AP to determine if the frame was successfully sent.

It can also optionally use Request to Send / Clear to Send signalling to improve performance.
