One solution to the tradeoff between simplicity and robustness that is inherent in centralised vs decentralised systems is to decentralise(ish) the leadership!

We want to choose a leader to undertake special tasks, like co-ordination.

When the leader fails, how do we detect, or recover from this?

In addition, all processes have to agree on the choice of leader.

For such a system, we want the following correctness properties to hold:

  • safety - a process has either not elected anyone or elected the non-crashed process with the largest known identifier
  • liveness - all processes participate and eventually choose some process to elect (or crash)

Also, we want to minimise the number of messages sent. This includes turnaround, which is the number of serialised transmissions in a single run.

A ring-based algorithm

This involves processes arranged in a logical ring.

Each process must have an identifier - there does not exist a deterministic algorithm for an anonymous ring leader election.

This is because every process would do exactly the same thing at exactly the same time or something.

This identifier must also be orderable. The highest number identifier gets picked to be the next leader.

Each process knows who its successor is and sends messages to them. Each process receives from its predecessor.

I have a feeling that they have to know more - each process has to know the entire ordering of the ring, otherwise if a single process went down it could no longer function.

It‚Äôs asynchronous.

  1. When a process notices that the coordinator is down, it builds an election message containing its own process identifier and sends the message to its successor.
  1. The successor adds its own identifier to the message and passes it on, effectively making itself a candidate to be elected as coordinator.
  1. Eventually, the message gets back to the process that started it. That process recognises this because it receives a message containing its own identifier.
  1. At that point, the message type is changed to ‚Äúcoordinator‚Äù and circulated once again to inform everyone else who the coordinator is (and who the new members are).
    1. The coordinator is the process whose ID has the highest value in the message.
  1. When this message has circulated once, it‚Äôs removed and everyone goes back to work.

Correctness

  • Safety - yes. Only the process with the highest ID can be elected as coordinator.
  • Liveness - yes. Every process in the ring eventually participates in the election and receives the ‚Äúcoordinator‚Äù message. Extra elections are stopped.

If a process fails after receiving but before sending, then we might be a tad fucked.

Bully election algorithm

Assumptions:

Processes can crash, but the message delivery is reliable.

Synchronous - we use timeouts to detect process crashes.

Each process knows all other processes and can communicate with them.

Each process knows which processes have higher identifiers.

  1. Any process, after detecting the failure of the leader, bids to become the new leader by sending an election message to every process with a higher identifier.
  1. If any process with a higher ID responds with an answer message, then the requesting process gives up its bid to become the leader. It then waits to receive a coordinator (or leader) message from some higher process.
    1. If this waiting takes too long, it will re-initiate the election because it assumes that the higher process failed in the meantime.
  1. If no higher-numbered process responds to the election message, then it elects itself the leader and sends a coordinator message to every process in the system.

Correctness

  • Safety - almost, but not quite.
    • A lower-id process always yields to a higher one, but if a failed process is replaced during an election, the lower-id processes could disagree on which is the correct leader.
    • Failure detection could be unreliable because of this.
  • Liveness - yeah. All the processes participate and know the coordinator at the end.
