MQTT and CoAP

MQTT

MQTT is a protocol designed to reduce overhead compared to HTTP. It’s a standard designed for Internet of Things applications.

In MQTT, messages are published to a broker.

Clients then subscribe to certain data streams from a broker.

These topics are hierarchical.

/house/room/main-light

# is a wildcard. So, /house/room/# will subscribe to everything in the room.

CoAP (Constrained Application Protocol)

CoAP was made for low-power systems - internet of things, for example.

It’s RESTful (uses GET, PUT, POST, etc).

This makes for easy web interoperability.

It also copies response codes from HTTP (404 etc)

It also has a low memory footprint, as is often required in such devices.

The basic packets should fit in a single 802.15.4 radio frame.

It’s a more modern, lightweight design than MQTT.

It typically uses UDP and the protocol format is binary to save on space.

Dealing with packet loss

CoAP uses confirmable requests.

If the client doesn’t receive an ACK back, they will attempt to resend the request.

Responding with the data on a constrained system can be slow, so the ACK gets sent immediately and the payload later.

A message ID is used to keep track of responses.

Proxies

Since these devices are limited by nature, proxies can handle requests to them to prevent overwhelming them / requesting when they’re down.

In addition, proxies can cache values, allowing nodes to sleep and conserve power.

Resource discovery

Use GET .well-known/core to receive a CoRE description of the resources on the node.

This is a standard format for defining resources.

This means that IoT objects can be added and their resources discovered automatically.
