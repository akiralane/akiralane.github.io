Modelling interaction

For distributed algorithms, behaviour and state are determined by the intertwining of the sequential steps of each involved process.

This means that a distributed algorithm typically (always?) involves the transmission of messages.

No one component of a distributed algorithm can have full, certain knowledge of global state (in a non-trivial system), since messages take time to pass.

Performance is important, obviously. There are the usual message-passing culprits:

  • Latency, measured from start of transmission to start of reception.
  • Bandwidth, the total amount of information able to be transferred over a given time.
  • Jitter, the variation of latency.

Synchronous vs Asynchronous

In a synchronous system, there are known upper and lower bounds on processing time.

Further, each message is received in a known bounded time and drift rate has a known bound.

These are not just properties - this is the definition (apparently https://www.cs.usfca.edu/~srollins/courses/cs682/web/notes/timeandstates.html.bak) of a synchronous distributed system.

Asynchronous systems, in contrast, have none of these guarantees.

They‚Äôre harder to think about and design, but they better model real-world usages.

Any solution which is valid here is also valid for a synchronous one.

Modelling Failure

The crash of a process is known as a process omission failure.

In a synchronous model, we can detect that a process has crashed by using a timeout.

The same can‚Äôt be said for asynchronous models - maybe the process is not replying because it‚Äôs slow or the message hasn‚Äôt arrived yet. There aren‚Äôt any bounds we can rely on.

The loss of a message is known as a channel omission failure.

Dealing with failure

We can use masking:

Replicate processes to tolerate process omission failures (crashes).

Replicate messages to tolerate channel omission failures (message loss).

We can convert the type of failure to a more useful one:

Use a checksum to detect corrupted messages and discard them. This converts an arbitrary (or byzantine) failure into a channel omission one.

Validity and Integrity

Timeout-based retransmission can guarantee validity farily easily.

Checksum + retranmission can guarantee the first part of integrity (but not the second).

We can just discard duplicate messages, though. This could be done through a message ID or timestamp system.
