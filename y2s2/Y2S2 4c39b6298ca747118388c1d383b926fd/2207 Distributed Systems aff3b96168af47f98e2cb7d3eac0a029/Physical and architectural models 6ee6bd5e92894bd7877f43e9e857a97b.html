Physical models

The baseline physical model is that:

There are components located at computers.

The computers are interconnected by a network.

The components communicate by passing messages over said network.

Architectural models

The architecture of a distributed system describes its structure in terms of the types of components involved and their relationships and interactions.

There are two main aspects to consider:

Elements - the building blocks, including things like components, methods of communcation, roles and deployment.

Patterns - common strategies for combining the elements.

Communcation paradigms

Inter-process communication (network level)

Message-passing and sockets.

Remote invocation (communcation protocols based on connection and exchange)

This includes things like request-reply protocols - HTTP.

Alternatively, Remote Procedure Call (RPC), where you request the execution of a specific remote function and receive the response.

Alternatively alternatively, you can be more specific and use Remote Method Invocation (RMI), which matches it with the problem-oriented communcating objects.

You can think of RMI as object-oriented RPC. It‚Äôs the invocation of a method of a remote object.

These approaches are limited by time and space coupling.

Senders need to know how to reach the receivers - the endpoints.

Both senders and receivers need to be up and running at the same time.

One approach is indirect communication - to decouple time and space between participants.

This includes things like group communcation, publish-subscribe systems, message queues, distributed shared memory etc.

Roles and responsibilities

Client-server

Clients send requests and receive responses back.

Servers can be clients of other servers.

Peer-to-peer

All involved entities play similar roles.

Same program, same interfaces, different machines.

Enables decentralised approaches, potentially improving scalability.

The design is often much more complex than client-server.

Placement

How do you map identified elements onto the underlying physical infrastructure?

For example, you can:

replicate data over multiple servers

  • Fault tolerance, load balancing
  • Need to ensure data consistency

partition data over multiple servers

  • Lack of real fault tolerance
  • No overhead from attempting to ensure consistency

Architectural patterns

Layering

This is the vertical organisation of services into layers.

Each layer provides a software abstraction and hides the details of the lower layers.

The middleware aims to mask the heterogeneity and provides convenient programming abstractions.

Tiering

Organise functionality of a given layer and place this functionality into servers.

Tiering is complementary to layering - it‚Äôs not an alternative.

For example, tiering a client-server application into presentation logic, application logic and data logic.
