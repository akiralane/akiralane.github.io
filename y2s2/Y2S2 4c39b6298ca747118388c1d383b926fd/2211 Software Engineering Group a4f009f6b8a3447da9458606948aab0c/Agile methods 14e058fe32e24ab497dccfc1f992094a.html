I already wrote down the XP practices but here they are again.

  • The planning game
    • User stories, captured on CRC story cards. CRC stands for Class Responsibility Collaboration, apparently.
  • Customer communication
    • Development team has continuous access to a live customer - someone who will actually be using the system.
  • Test-driven development
    • Tests come before functionality. Functionality is written to meet tests and no more.
  • Pair programming
    • ew
  • Small releases
    • Start with the smallest useful feature set. Release early and often.
  • Continuous integration
    • Developers integrate new code into the system as often as possible. All functional tests must still pass after integration or the new code is rolled back.
  • Design improvement (previously called refactoring)
    • As developers work, the design should be continually evolved to keep it as simple as possible, with no unneccessary code.
  • Collective ownership
    • The code is owned by all developers, and they may make changes anywhere in the code at any time they feel necessary.
  • Metaphor
    • Customers, managers and developers construct a metaphor or set of metaphors after which to model the system.
  • Simple design
    • Developers ‚Äúsay everything once and only once‚Äù in their design.
  • Coding standards
    • A team must have common sets of coding practices, using the same formats and style for code writing.
  • Sustainable pace (used to be called 40-hour week)
    • Scope and prioritising, avoidance of burnout
