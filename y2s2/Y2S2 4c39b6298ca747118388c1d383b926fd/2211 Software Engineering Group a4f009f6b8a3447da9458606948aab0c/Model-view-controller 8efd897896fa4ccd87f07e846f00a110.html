What problem does the model-view-controller thing solve?

Technical design outlines have high coupling, low cohesion and no separation of concerns.

This means that it is difficult to:

  • Understand
  • Extend
  • Re-use
  • Test

The MVC pattern

The model contains domain data.

The view renders data for display.

The controller mediates between model and view.

Model

The model models the domain of the application.

This is usually a one-to-one mapping to real-world objects or concepts.

It contains relevant domain or business logic, responding to state queries and operations on its state.

View

The view graphically represents the state of the model to the user.

It can consist of many sub-views.

It’s typically passive - it responds to update operations.

In this fashion, it receives user input, but passes this on to the controller.

Controller

This is the plumbing between the model and the view.

It translates user actions (button presses, keyboard input) into operations on the model.

It may be very simple - a single class is sometimes acceptable, but decomposition and multiple controllers for different sub-views or functionalities are possible.

Benefits of MVC

Clear separation of concerns.

Low coupling.

Easy to maintain - especially view.

Easy to test.

Promotes re-use.

Can support multiple view implementations independent of the model.

MVC implementations

The passive model pattern is where the model responds to state change operations but the controller is responsible for notifying the view.

On the other hand, the active model pattern is where the model responds to state change operations and notifies view of the change using the observer pattern.

The iOS MVC is an extension of the active model. The controller passes on updates to the model.
